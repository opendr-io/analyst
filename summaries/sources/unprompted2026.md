## Major Themes And Trends

### 1. Agentic AI is treated as the new default security substrate

The records repeatedly assume that security teams are not merely using chatbots, but integrating agents into operational workflows. OpenAI’s “Code Is Free” frames LLMs as already writing engineers’ code and asks why they are not also used for security invariants and low-friction security improvements [record_id:2325]. Stripe describes agents in production for threat modeling and security request routing, emphasizing modular orchestrator/child architectures, structured I/O, and evaluation loops [record_id:2324]. Adobe describes an AI-native security guidance service powering multiple defensive AI capabilities from centralized evaluated knowledge [record_id:2323]. Salesforce’s agentic SOC talks argue that defense is moving beyond Q&A copilots toward autonomous agents that “plan, reason, and act,” implemented through a supervisor-worker architecture [record_id:2380] [record_id:2381].

The trend is not simply more automation; it is deeper integration into security workflows. Datadog’s threat-hunting talk describes automating hypothesis-driven query generation, iterative refinement, and narrowing toward pivotal evidence [record_id:2364]. GreyNoise’s Orbie operates over internet-scale honeypot data to surface emergent threats, identify campaigns, and write detection rules [record_id:2365]. Rob T. Lee’s SIFT talk connects Claude Code to a DFIR workstation through Model Context Protocol to support timeline generation, memory analysis, and malware sweeps [record_id:2340]. Rami McCarthy’s Shai-Hulud postmortem describes moving from simple “vibe-coded” scrapers to multi-agent triage engines for victimology and secret-impact analysis [record_id:2332].

### 2. Measurement, evaluation, and verification are central concerns

A major theme is that naive benchmarks and output-only scoring are inadequate for real-world security agents. Joshua Saxe’s talk explicitly says the hard part is not building agents, because AI coding tools drive agent-building costs toward zero; the hard part is knowing whether they work against real attacks and unseen vulnerabilities [record_id:2322]. The abstract describes a journey from basic precision/recall on old data toward multi-dimensional evaluation covering reasoning quality, evidence gathering, and tool-calling logic [record_id:2322].

Mudita Khurana similarly argues that security-agent evaluation is still rooted in narrow, outcome-only benchmarks and should instead assess end-to-end workflows across find, confirm exploit, patch, and validate loops [record_id:2360]. Stripe’s production-agent abstract also emphasizes offline and online evaluation loops, including the challenge of subjective outputs in threat modeling compared with higher-signal feedback in routing [record_id:2324]. FENRIR uses a multi-stage verification pipeline with static-analysis pre-triage, two-layer LLM validation, and confidence-based human routing, explicitly distinguishing what worked from what failed [record_id:2329]. NVIDIA’s Marinade injects realistic vulnerabilities into real code and ships each injected bug with a validation script to prove exploitability, addressing false-positive and benchmark-quality concerns [record_id:2344].

This collection suggests a maturing research culture: rather than treating demos as sufficient, many speakers discuss evaluation harnesses, validation scripts, latency constraints, confidence routing, production feedback, and benchmark realism [record_id:2322] [record_id:2329] [record_id:2344] [record_id:2360] [record_id:2363] [record_id:2377].

### 3. Prompt injection remains unresolved, but containment and authorization are emerging as practical defenses

Prompt injection is presented as a persistent and structurally difficult problem. Andrew Bullen’s Stripe talk states directly that “there’s no deterministic defense” to prompt injection, but argues for containment: controlled egress, human confirmation flows, and CI-time enforcement using tool annotations [record_id:2358]. Google Workspace’s talk frames the “Perfect Storm” as the intersection of sensitive data access, untrusted content exposure, and external command execution, and discusses defense-in-depth for Gemini and Workspace, including a calendar-invitation hijack example, chaining policies, sandboxing, and sanitization [record_id:2361].

Several talks explore more formal or deterministic controls around agent actions. Matt Maisel proposes a reference monitor using Rust hooks and Cedar policies to intercept shell commands, file reads, and other actions by coding agents, with deterministic policy enforcement around exfiltration and destructive behavior [record_id:2345]. Niki Aimable Niyikiza proposes capability-based authorization through ephemeral cryptographic warrants inspired by Macaroons/UCAN, task-scoped and holder-bound, so even a compromised agent cannot exceed its delegated authority [record_id:2370]. Ilia Shumailov’s abstract is brief but claims methods for running modern AI agents with security guarantees, including complex computer-use setups [record_id:2378].

The records also show attacks evolving beyond simple prompt strings. Sean Park describes document-embedded injection in AI-driven KYC pipelines, where passports and regulatory verification workflows become part of an exploit chain for cross-record reads/writes and exfiltration [record_id:2328]. Johann Rehberger’s “promptware” talk describes engineered prompts functioning like malware, with delayed tool invocation, persistence, lateral movement, C2, and data exfiltration [record_id:2369]. Kyle Polley’s BrowseSafe talk notes that linguistic camouflage, multilingual instructions, and visible text placement were challenging for production browser-agent prompt-injection detection [record_id:2363]. Joey Melo’s Parseltongue Protocol studies 100+ encoding and encryption techniques across 17,000 malicious prompts, highlighting multilingual and multi-encoding attack surface [record_id:2375].

### 4. AI is accelerating vulnerability discovery and offensive capability

Many talks claim significant advances in AI-assisted or AI-autonomous vulnerability discovery. Nicholas Carlini states that large language models can automate attacks that previously required human adversaries and that recent state-of-the-art models can find zero-day vulnerabilities in large software projects tested by humans for decades [record_id:2334]. AISLE’s OpenSSL talk claims an AI system found 12 vulnerabilities fixed in a January 2026 OpenSSL security update, including three that had existed for over two decades, and says AI has changed vulnerability-discovery economics [record_id:2350]. FENRIR claims 100+ vulnerabilities across AI infrastructure since mid-2025, with 21 CVEs patched and multiple CVSS 9.8 RCEs [record_id:2329]. Promp2Pwn claims an agentic AI found a vulnerability in Samsung’s Bixby for Pwn2Own [record_id:2357].

The methods vary. XBOW’s “Auth-by-One” talk focuses on AI agents for finding and validating authentication and authorization flaws, using strict validators to determine successful login or protected-resource access [record_id:2326]. Netflix’s “Source to Sink” discusses improving LLM first-party vulnerability discovery by reducing noisy findings and making an agentic pipeline “think before it screams” [record_id:2374]. Olivia Gallucci describes using AI agents with Apple’s partial open-source codebase for reverse engineering, fuzzing, diff triage, and prioritizing fuzzing targets [record_id:2356]. Jenny Guanni Qu’s kernel-bug talk presents lessons from training on 125K Linux kernel commits, including counterintuitive findings such as “hard negatives” hurting performance and subsystem boundaries hiding bugs [record_id:2376].

The evidence in these abstracts is strong in the sense that multiple independent organizations report similar trends, but thin where raw records omit details. The strongest claims—12 OpenSSL zero-days, 21 CVEs, 100+ vulnerabilities, 2.5x over CodeQL, 0-days in decades-old code—require follow-up into the videos, patches, CVEs, and papers for verification [record_id:2329] [record_id:2350].

### 5. AI-assisted development creates new AppSec and endpoint attack surfaces

Several records focus on AI IDEs, coding agents, and “vibe coding.” Piotr Ryciak’s talk says AI IDEs and coding agents expand the attack surface by creating paths from untrusted workspace inputs to high-impact actions, with exploitation patterns across OpenAI Codex, Amazon Kiro, Google Antigravity, Cursor, and others [record_id:2335]. The abstract organizes attacks by zero-click, one-click, autorun, and time-delayed execution [record_id:2335]. Srajan Gupta describes vibe coding as fast but as quietly bypassing traditional AppSec controls, and proposes an MCP server that injects threat models, security requirements, and OWASP guidance before code generation, then verifies generated output [record_id:2373]. OpenAI’s talk advocates engineering-first, low-friction security additions in an agentic future where “code is free” [record_id:2325].

Endpoint and observability concerns appear alongside developer-tool security. Elastic’s talk asks whether defenders can see what AI saw, focusing on telemetry for AI-executed commands, AI-spawned process ancestry, blind spots, and OpenTelemetry semantic conventions for GenAI endpoint activity [record_id:2341]. Matt Maisel’s Cedar-policy reference monitor also directly addresses coding agents’ access to code and terminals [record_id:2345]. Brooks McMillin’s Dropbox talk discusses daily-driver agents and what breaks in practice: capability bounding, prompt-injection detection, multi-agent memory isolation failures, and OAuth device flow for headless operation [record_id:2359].

Together, these records show AppSec shifting left into the AI coding loop but also shifting down into runtime policy enforcement, terminal mediation, and endpoint telemetry.

### 6. Enterprise governance is moving from abstract frameworks to operating models

Governance talks repeatedly emphasize balancing adoption and risk rather than banning AI. Billy Norwood discusses a risk-based AI governance committee in a healthcare services firm and lessons from successes and failures [record_id:2337]. Snowflake’s enterprise governance talk emphasizes governance frameworks, cross-functional collaboration, and embedding security into emerging technologies [record_id:2338]. Chase Hasbrouck describes three phases of enterprise AI adoption in Army Cyber: fragmented research previews, centralized token-budgeted solutions that inhibited experimentation, and enterprise agreements that removed cost barriers but introduced cultural-change challenges [record_id:2339].

Maxim Kovalsky’s “AI Security Larsen Effect” argues that existing guidance such as OWASP and NIST identifies what can go wrong or how to govern, but not which risks matter for a specific architecture or how to implement controls with an existing stack [record_id:2347]. That talk proposes a capability-based framework to decide whether to configure, buy, or build controls, demonstrated against a healthcare chatbot with PHI and an Azure/CrowdStrike stack [record_id:2347]. Joe Sullivan’s AI notetaker talk expands governance into everyday meeting infrastructure, noting risks around shadow deployment, vendor fragility, consent liability, discovery exposure, and the role of AI summaries as official records [record_id:2330].

The governance theme is practical and organizational: adoption constraints, budgets, culture, accountability, procurement, policy, and lifecycle management matter as much as technical controls.

### 7. Detection engineering is adapting to semantic, behavioral, and agentic signals

Traditional security detection appears insufficient against GenAI-era threats, but several talks propose adaptations. Microsoft’s BinaryShield addresses cross-service threat intelligence for prompt injection without sharing raw prompts, using privacy-preserving fingerprints [record_id:2327]. Palo Alto Networks’ SYARA extends YARA-like rule syntax with semantic detection using strings, embeddings, ML classifiers, and LLMs, aiming for low-latency, low-cost layered detection [record_id:2342]. Salesforce’s Agentforce defense uses behavioral baselines such as Asset Rarity and Query Complexity to reduce millions of prompts and 12,000 autonomous agents into fewer than 30 daily high-fidelity alerts [record_id:2377].

Detection also extends to AI-assisted attacker behavior. Sergej Epp’s “8 Minutes to Admin” describes two AI-assisted campaigns: an 8-minute AWS escalation from stolen credentials to full admin and EtherRAT, a fileless Node.js implant using Ethereum smart contracts for C2 [record_id:2355]. The abstract notes that these campaigns did not introduce novel primitives but compressed known techniques to speeds and scales that break traditional detection models [record_id:2355]. GreyNoise’s Orbie uses internet-scale honeypot data to catch campaigns missed by traditional methods [record_id:2365]. Datadog’s internal threat-hunting automation defines an “automation boundary” around where AI accelerates work versus introduces risk [record_id:2364].

The trend is toward richer telemetry, semantic matching, behavioral baselines, privacy-preserving sharing, and human-trust design rather than purely signature-based detection.

### 8. AI security is broadening beyond software into infrastructure, hardware, and physical systems

The source includes talks on confidential AI, hardware hacking, and physical AI. Raghu Yeluri’s confidential AI talk describes hardware-based protection for data and models during active processing using TEEs across CPUs, GPUs, and accelerators, plus remote attestation for public and hybrid cloud deployments [record_id:2343]. Adam Laurie’s hardware talk claims Claude was given control of a hardware lab—including USB hub, XYZ platform, PICO2, Jlink-pro, oscilloscope, Chipshouter, and targets—and succeeded at glitching an LPC chip within minutes after weeks of human failure [record_id:2331]. Padma Apparao’s physical AI talk argues that when AI enters robotics and autonomous machines, failure shifts from misinformation to kinetic damage, requiring system-level risk thinking around VLA models, sensor spoofing, embodied instruction manipulation, and governance gaps [record_id:2348].

These talks mark a widening threat model: AI agents are not only text processors or code assistants; they increasingly interact with clouds, terminals, hardware labs, sensors, robots, and business processes.

## Gaps, Limits, And Open Questions

The largest limitation is that these records are abstracts, not full papers or transcripts. They frequently make strong claims—zero-days found, detection rates achieved, vulnerabilities patched, production deployments operating at scale—but provide limited methodological detail in the raw text. For example, FENRIR’s claim of 100+ vulnerabilities and 21 CVEs [record_id:2329], AISLE’s claim of 12 OpenSSL zero-days [record_id:2350], BrowseSafe’s F1 score and latency claims [record_id:2363], and SYARA’s 98% detection under 100ms and $0.001/query claim [record_id:2342] are important but need independent validation.

Another gap is comparative evaluation. Many talks present a tool or architecture, but the records do not consistently describe baselines, failure rates, adversarial robustness over time, or reproducibility. Some abstracts do mention what failed—FENRIR says pure automation, generic prompts, and insufficient context failed [record_id:2329]; Jenny Guanni Qu says obvious ML vulnerability-detection approaches and hard negatives failed in kernel-bug detection [record_id:2376]; Datadog discusses hallucinations and trust tradeoffs [record_id:2364]. But many records do not provide enough detail to compare approaches.

Prompt-injection defense remains unresolved. Several records propose containment, policy enforcement, browser-agent detection, semantic detection, capability tokens, and guarantees [record_id:2358] [record_id:2345] [record_id:2363] [record_id:2370] [record_id:2378]. Yet the abstracts collectively imply that no single solution is definitive. Open questions include how to compose defenses across agents, tools, identity systems, browsers, documents, endpoints, and human approval workflows; how to verify the correctness of capability constraints that depend on runtime context; and how to prevent insecure defaults from recurring across new agents [record_id:2369] [record_id:2370].

Governance evidence is broad but not deeply empirical in the raw text. Records describe committees, frameworks, adoption phases, and enterprise balancing acts [record_id:2337] [record_id:2338] [record_id:2339] [record_id:2347], but do not provide enough detail to assess outcomes across organizations. More research is needed on which governance models measurably reduce AI incidents without suppressing useful adoption.

The physical AI and hardware records broaden the field but are relatively isolated. Confidential AI [record_id:2343], hardware hacking with AI-controlled lab equipment [record_id:2331], and kinetic risk in robotics/VLA systems [record_id:2348] raise major implications, but the collection has fewer talks in these areas than in software agents and detection. Future research should connect agent containment and authorization ideas to hardware, robotics, and safety-critical environments.

The collection also has limited disagreement. Most records share an implicit pro-adoption stance: AI is risky but unavoidable, and defenders must learn to use it. There are cautions about “lazy” AI [record_id:2332], unreliable benchmarks [record_id:2360], insecure defaults [record_id:2369], and governance gaps [record_id:2330] [record_id:2348], but few records argue for slowing deployment or rejecting agentic approaches outright. Downstream researchers should therefore be alert to selection bias: the conference source likely overrepresents builders and early adopters.
