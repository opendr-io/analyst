# Topic: Source: promptorgtfo

## Executive Summary

The `promptorgtfo` source consists of 67 records from the Prompt||GTFO event series, all dated 2026, centered on practical demonstrations of AI use in security, software development, research, governance, productivity, and creative work. The collection reads less like a conventional academic corpus and more like a practitioner showcase: speakers demonstrate working prototypes, live workflows, experiments, and failure modes involving LLMs, agents, MCP integrations, AI coding tools, workflow automation platforms, and security-specific AI systems.

The dominant pattern is pragmatic experimentation. Many records show security professionals using AI to accelerate existing work: threat intelligence, SOC triage, reverse engineering, vulnerability management, compliance mapping, audit evidence collection, research synthesis, interview preparation, and coding. At the same time, the series repeatedly emphasizes that AI systems are brittle, error-prone, and attackable. Several talks focus on prompt injection, jailbreaks, system prompt leakage, malicious MCP servers, AI agent compromise, unsafe generated code, guardrail evasion, and LLM hallucination or reasoning failures [record_id:2179] [record_id:2188] [record_id:2199] [record_id:2207] [record_id:2208] [record_id:2209] [record_id:2211] [record_id:2212] [record_id:2230] [record_id:2237] [record_id:2243].

A recurring conclusion across the records is that AI is most valuable when embedded into constrained workflows with validation, human oversight, structured outputs, deterministic fallback logic, source traceability, or model cross-checking. Talks on JSON validation, threat intelligence extraction, Splunk-to-Sigma conversion, city council transcript summarization, CodeQL-based cryptographic mapping, and requirements engineering all show attempts to impose structure around LLM output [record_id:2186] [record_id:2193] [record_id:2196] [record_id:2204] [record_id:2227] [record_id:2233]. Conversely, records describing fully automated research pipelines, embedded firmware coding, and AI-generated code security highlight persistent limitations: hallucinated details, false positives, fabricated statistics, irrelevant citations, insecure defaults, and poor understanding of real-world systems [record_id:2179] [record_id:2185] [record_id:2206] [record_id:2211].

Overall, Prompt||GTFO appears to document an emerging practitioner culture around “AI as a power tool”: useful, fast, and democratizing, but requiring adversarial thinking, guardrails, instrumentation, and skepticism.

## Major Themes And Trends

### 1. AI as an accelerator for security operations, but not a replacement for judgment

Many talks show AI compressing repetitive analyst work into faster, more structured workflows. Jeremy Snyder describes using LLMs for log analysis, data classification, and anomaly detection, with practical lessons around separating DLP and anomaly prompts, exploiting large context windows, batching, and prompt caching for cost control [record_id:2190]. Jonathan Cran demonstrates structured-output threat intelligence tooling that extracts entities from webpages and PDFs, generates Sigma rules from exploit analysis, and uses Langchain/Langgraph with VirusTotal and URLScan for phishing URL analysis [record_id:2204]. Rick Deacon’s N8N threat-intelligence workflow dynamically generates company-specific searches, scrapes sources, stores findings, and produces digests with voice synthesis for Slack [record_id:2226]. David Kennedy’s Night Beacon uses a locally running fine-tuned BERT model to assist SOC analysts by assessing alerts, prioritizing risk, integrating into SOAR, and learning from analyst feedback [record_id:2240].

A related group of records applies AI to investigation and detection engineering. Leo Meyerovich’s “vibes investigating” uses generic markdown templates and Claude Code agents to solve Splunk Boss of the SOC and Kusto Detective Agency challenges quickly, with cross-validation built into plan files [record_id:2222]. Justin Borland’s Splunk-to-Sigma work is especially important because it shows a transition from using an LLM as the conversion engine to using the LLM to build a deterministic parser, yielding faster and more reliable conversion [record_id:2233]. Ryan Moon’s phishing and social-engineering detection approach uses concise yes/no label prompts to supplement traditional tools like Snort, YARA, and PCRE when attacks lack obvious technical indicators [record_id:2244]. Gadi Evron’s linguistic analysis of a Hebrew phishing email uses ChatGPT to infer possible source-language interference and dialect cues, positioning LLMs as triage aids rather than definitive attribution tools [record_id:2235].

Across these records, the trend is not “AI replaces the SOC,” but “AI narrows the analyst’s search space.” Stronger records emphasize structured output, cost control, deterministic validation, analyst feedback, and bounded tasks [record_id:2190] [record_id:2204] [record_id:2233] [record_id:2240] [record_id:2244].

### 2. AI agent and LLM security is treated as application security, not only model behavior

Several records argue, explicitly or implicitly, that the security boundary is the full AI application stack: tools, prompts, APIs, memory, browser extensions, sandboxes, MCP servers, data access, and orchestration logic.

RSnake’s LLM detection talk uses refusal patterns and “flow breaking” to detect LLM-mediated systems and stresses that red teaming must attack the whole application rather than just the base model [record_id:2188]. The episode 5 general chat similarly discusses prompt extraction, LLM detection, flow-breaking attacks, and API keys exposed in system prompts [record_id:2187]. Shlomo Touboul’s malicious MCP demonstration shows how a tool server can inject hidden metadata instructions that manipulate Claude Desktop’s reasoning and actions, including misdirecting a revenue-reporting workflow and secretly writing files during a weather-tool interaction [record_id:2207]. Rene Brandel’s report on hacking 16 YC Spring Batch AI agents found 7 compromised in about 30 minutes each, with patterns including IDOR via extracted tool IDs, weak code sandboxes, cloud metadata exposure, and SSRF through tool parameters leaking git credentials [record_id:2209].

These records suggest a shift from classic prompt-injection demos toward AI application exploitation: agent tools expose IDs, sandboxes execute code, MCP servers become trust boundaries, and prompts may contain secrets. The collection repeatedly frames AI systems as software systems with familiar web, cloud, and application-security failures, amplified by LLM-specific trust and instruction-following issues [record_id:2187] [record_id:2188] [record_id:2207] [record_id:2209].

### 3. Guardrails, prompt patching, jailbreaks, and refusal behavior are recurring concerns

A cluster of talks examines adversarial prompting and safety mechanisms. Itsik Mantin’s “prompt patching” introduces “security steerability,” measuring how well models follow system-prompt restrictions despite adversarial user instructions, and compares model behavior across GPT-4.1 variants and Gemini Flash [record_id:2199]. Michael Bargury’s chatbot.ai allows testers to run the same jailbreak prompt across many models and configurations, helping determine cross-model transferability [record_id:2212]. Aur Saraf’s protocol confusion attack exploits a mismatch between chatbot state and guardrail state, using encoded content and conversation replay to bypass controls at a conceptual level [record_id:2243]. Dragos Ruiu demonstrates multi-model orchestration for extracting safety filtering rules and guidelines via tokenization-boundary manipulation, reflection, and context-window shifting, while also noting cross-model validation for code generation [record_id:2237].

Other records examine refusal and dangerous-output boundaries. Nathan Case discusses how AI models can be manipulated through pretexting to produce hazardous bioweapon-related content, with newer models showing stronger but still bypassable safeguards in his reported experiments [record_id:2208]. Ronald Gula’s Vyvron avatar demo includes the avatar successfully refusing a jailbreak attempt involving illegal drug manufacturing queries, making refusal behavior part of an AI avatar trust demonstration [record_id:2232]. The closing discussion of Prompt||GTFO #1 also covers prompt injection versus jailbreaking concepts, context-window management, modular code generation, and iterative patience [record_id:2234].

Collectively, these records show a practitioner community trying to move from anecdotal jailbreaks toward repeatable testing, benchmarks, patching, and multi-model comparison. However, much of the evidence remains demo-based rather than standardized evaluation [record_id:2199] [record_id:2212] [record_id:2237] [record_id:2243].

### 4. AI coding is powerful, but brittle without architecture, tests, constraints, and review

The records present both enthusiasm and skepticism about AI-assisted development. Imri Goldberg’s game-development project with his 8-year-old son uses Godot, Claude Code, Cursor, and related tools to “vibe code” a space shooter, while emphasizing architecture, testing, and modular design to keep AI-generated code stable [record_id:2183]. Gadi Evron’s Mail Goggles Chrome extension shows iterative development with GPT-5, specialized developer/tester prompts, and challenges around UI bugs and Manifest 3, including a partially failed live demo that eventually worked [record_id:2189]. His simple mail merge demo shows Claude generating a Google Apps Script for Gmail/Sheets with personalization, deduplication, test sends, and randomized pacing [record_id:2203]. Gadi’s broader “AI Automation for Every Employee” talk frames Claude-generated Google Apps Scripts as a way to empower non-programmers to automate spreadsheet workflows without learning Python [record_id:2214].

Other software-development records show sharper limits. Ryan Torvik’s embedded firmware experience is one of the clearest negative examples: multiple tools hallucinated board support packages, copied files incorrectly, failed to compile/link, and could not reason about real hardware; success came only after manual STM32 project setup and constraining the AI to small insertions between comment markers [record_id:2206]. Chris Wysopal’s Veracode study reports that syntax quality has improved while security pass rates remain flat; LLMs can identify and fix vulnerabilities when asked, but do not write secure code by default [record_id:2211]. The episode 4 AI coding discussion debates whether vibe coding will become default, mentions research that AI may slow experienced developers, and discusses practical use of Cursor and Claude Code [record_id:2201].

A nuanced positive example is Halvar Flake’s PyTorch optimization demo: Claude Code and Gemini CLI helped profile and rewrite a bottlenecked neural-network visualization, moving hashing into GPU-side forward hooks and improving performance from roughly a minute per frame to about half a second [record_id:2223]. Allan Stojanovic’s requirements-engineering Gemini Gem shows using AI before coding: asking one question at a time, tracking conflicts, reducing chattiness, and producing formal requirements documents for later engineering work [record_id:2227]. Brandon Helms’ Dallas Debugger shows a secure Kubernetes debugging tool built through prompt engineering with Cursor, explicitly designed around CIS benchmarks and security-by-design constraints [record_id:2218]. Shira Luk-Zilberman’s Base44 demo shows AI-generated hosted web applications from prompts, including event registration and database backend functionality [record_id:2242].

The trend is clear: AI coding works best when the problem is bounded, the environment is well-instrumented, and humans impose architecture, tests, or constraints. It struggles where correctness depends on hidden physical state, secure defaults, or complex integration details [record_id:2206] [record_id:2211].

### 5. MCP becomes both an integration pattern and a threat surface

Model Context Protocol appears repeatedly as a way to connect LLMs to tools, files, security products, and analysis environments. Positive MCP use cases include Rich Mogull’s Claude-controlled Z5 game player and MCP server for a CTF-style Infocom game [record_id:2200], Joshua Reynolds’ Binary Ninja MCP server for automated malware analysis and detection-rule generation [record_id:2228], Randy Pargman’s Ghidra MCP integration for reverse engineering a malware DLL [record_id:2236], and Fyodor Yarochkin’s Claude Desktop setup with Brave Search, Memory MCP, File System MCP, Passive DNS, VirusTotal, and AWS Athena for investigating AI-generated content online [record_id:2219].

Community discussion records also treat MCP as a practical architecture: Prompt||GTFO #3’s open floor covers MCP servers as REST API layers, self-hosted versus cloud models, cost optimization, debugging generated code, and local models for guardrail-bypassing security tasks such as malware deobfuscation [record_id:2213]. Pedram Amini’s workflow mentions custom MCP integrations with Granola, iMessage, and other tools as part of a command-line/Obsidian/Fabric productivity system [record_id:2231].

At the same time, MCP is framed as an attack surface. Shlomo Touboul’s malicious MCP server demonstrations show hidden instructions inside tool metadata manipulating Claude Desktop’s logic and actions [record_id:2207]. The same pattern is implicit in broader AI-agent security records where tools and permissions become exploitable system components [record_id:2209]. The corpus therefore treats MCP as a major enabling technology for AI workflows, but one that needs security review, least privilege, inspection, and trust-boundary design.

### 6. Structured outputs, deterministic parsing, and validation are repeated antidotes to hallucination

A cross-cutting lesson is that the best AI workflows convert fuzzy language-model output into structured, validated, auditable artifacts. RSnake’s JSON summarization system uses local Llama 3.2 with ChatGPT fallback to summarize thousands of geopolitics articles into validated JSON, with early token validation, markdown stripping, banned-word enforcement, and retry logic [record_id:2196]. Jonathan Cran’s threat-intelligence tooling uses OpenAI structured output and Pydantic objects to extract entities and generate detection artifacts [record_id:2204]. Heather Linn’s regulatory-compliance tool outputs downloadable JSON taxonomies with document topics, named entities, business intents, and data classifications linked to regulations [record_id:2239]. Daniel Cuthbert’s cryptographic-discovery project normalizes scanner output, uses CodeQL to extract crypto paths, and visualizes cryptographic bills of materials in graph form [record_id:2186].

The same pattern appears in research and summarization workflows. Brandon Keep’s YATSEE pipeline maintains timestamp traceability to source videos while using local Whisper/Faster Whisper and Ollama for multi-pass meeting summarization [record_id:2193]. Matthias Muhlert argues that high-quality source papers in NotebookLM reduce hallucinations compared with broad web search or less controlled RAG approaches [record_id:2217]. The grugq’s PhD research-assistant demo, however, shows that Claude can mis-select papers from Zotero and requires significant manual correction [record_id:2185]. Gadi Evron’s automated research-pipeline attempt similarly produced false positives, fabricated statistics, and systemic failures that required manual oversight [record_id:2179].

The evidence points to a shared operational doctrine: constrain the model, validate outputs, preserve source traceability, and avoid letting LLMs be the only truth source [record_id:2179] [record_id:2185] [record_id:2193] [record_id:2196] [record_id:2204] [record_id:2217] [record_id:2233].

### 7. Governance, compliance, policy, and executive communication are fertile AI use cases

The risks are less about exploit execution and more about accuracy, privacy, governance of connected data, and overconfidence in fluent outputs [record_id:2191] [record_id:2220] [record_id:2239].

## Notable Talks, Records, And Evidence

Several records stand out as particularly representative or important for downstream research.

Chris Wysopal’s “Do LLMs Dream of Secure Code?” is among the strongest empirical records because it summarizes testing 100 LLMs across four languages and four vulnerability classes. Its core finding—that syntax improved while security pass rates remained flat, and that models can fix vulnerabilities when asked but do not write secure code by default—is central to any assessment of AI coding risk [record_id:2211].

Rene Brandel’s “How We Hacked Y Combinator Spring Batch’s AI Agents” is a key AI application-security record. It moves beyond abstract prompt injection to concrete compromise of real AI agents through IDOR, sandbox escape/code execution paths, cloud metadata access, SSRF, and credential leakage, with 7 of 16 agents compromised quickly [record_id:2209]. Shlomo Touboul’s malicious MCP server talk is similarly important because it identifies MCP tool metadata as a manipulation channel that can redirect model behavior and hide tool actions from users [record_id:2207].

Itsik Mantin’s prompt patching talk matters because it offers a named evaluation concept—security steerability—and compares model families/strengths against adversarial instructions [record_id:2199]. Michael Bargury’s chatbot.ai complements this by providing multi-model jailbreak testing infrastructure [record_id:2212]. Aur Saraf’s protocol-confusion guardrail bypass and Dragos Ruiu’s multi-model prompt-leakage work broaden the adversarial landscape to state mismatch, encoding, reflection, tokenization boundaries, and context shifting [record_id:2243] [record_id:2237].

For operational security use, David Kennedy’s Night Beacon is notable because it describes an AI system already embedded in SOC triage through a small API surface and feedback loop, using a local model rather than only cloud LLMs [record_id:2240]. Jonathan Cran’s structured-output threat intel tooling is representative of practical LLM augmentation for entity extraction, detection-rule generation, and phishing URL analysis [record_id:2204]. Justin Borland’s Splunk-to-Sigma conversion is a valuable cautionary record: after LLM guardrails and validation, the better solution was a deterministic parser generated with AI assistance [record_id:2233].

For reverse engineering, Joshua Reynolds’ Binary Ninja MCP and Randy Pargman’s Ghidra MCP demos are central, showing LLMs directly accelerating malware analysis, IOC extraction, detection-rule generation, and database annotation [record_id:2228] [record_id:2236]. These records should be paired with malicious-MCP and AI-agent security records because the same integration pattern that accelerates analysis also expands the trust boundary [record_id:2207].

For AI-assisted research and knowledge work, Gadi Evron’s notetaker/research-pipeline talk and the grugq’s Claude research-assistant talk are important negative evidence. Both describe useful assistance but also false positives, fabricated statistics, irrelevant paper selection, and the need for manual oversight [record_id:2179] [record_id:2185]. Matthias Muhlert’s NotebookLM/Elicit workflow and Brandon Keep’s YATSEE pipeline offer contrasting mitigation patterns: feed high-quality sources, preserve traceability, and structure outputs [record_id:2217] [record_id:2193].

For governance, Gary Hayslip and Sarah Miller’s SID the CyberCat, Sounil Yu’s Neo, Guillaume Ross’s AI browser-extension inventory, Heather Linn’s compliance taxonomy generator, and Chris Inglis’s policy-paper drafting show how AI is being used for audit, compliance, vendor mapping, privacy review, and strategic policy work [record_id:2191] [record_id:2197] [record_id:2220] [record_id:2239] [record_id:2198].

## Gaps, Limits, And Open Questions

The most significant limitation is that most records are demo summaries, not reproducible studies. They often report impressive results—seconds instead of hours, 80% mapping accuracy, near-100% CTF success on easier challenges, successful agent compromises, large time savings—but the underlying datasets, prompts, benchmark methods, and failure distributions are not usually described in enough detail to generalize [record_id:2197] [record_id:2222] [record_id:2228] [record_id:2241]. Downstream researchers should treat these as leads and practitioner evidence, not as settled quantitative conclusions.

A second gap is evaluation methodology. The corpus repeatedly identifies hallucination, false positives, fabricated facts, and insecure defaults, but only some talks define validation approaches. Stronger methodological records include security steerability benchmarking, JSON validation, deterministic parser generation, structured-output extraction, and quantified LLM-generated-code testing [record_id:2196] [record_id:2199] [record_id:2204] [record_id:2211] [record_id:2233]. Many other workflows would benefit from explicit measurements of precision, recall, time saved, cost, false-negative rates, and human correction burden.

A third gap concerns safety boundaries and responsible deployment of offensive or dual-use systems. Records describe AI-powered phishing agents, offensive recon harnesses, bug-bounty payload generation, guardrail bypasses, prompt leakage, and hazardous-content manipulation [record_id:2182] [record_id:2192] [record_id:2194] [record_id:2208] [record_id:2237] [record_id:2243]. The summaries usually mention defensive implications or research context, but they do not consistently document safeguards, access controls, disclosure processes, or operational constraints.

A fourth open question is how enterprises should secure MCP and agent ecosystems. The records show both high-value MCP integrations and malicious MCP risks [record_id:2207] [record_id:2219] [record_id:2228] [record_id:2236]. Needed research includes MCP permissioning, tool metadata inspection, least-privilege design, provenance, runtime monitoring, safe file-system access, and user-visible action auditing.

A fifth gap is long-term maintainability.
