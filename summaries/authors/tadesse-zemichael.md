# Topic: Author: Tadesse Zemichael

## Executive Summary

The two records attributed to Tadesse Zemichael show a research profile centered on applying machine learning and AI techniques to operational cybersecurity problems. The records are both CAMLIS conference entries, one from 2022 and one from 2024, and they span two related but distinct areas: identity-based threat detection in Azure Active Directory sign-in logs, and automated vulnerability management for containerized systems using LLM agents.

Across the records, Zemichael’s work emphasizes reducing manual analyst burden by modeling complex security context more effectively. In the 2022 record, the focus is on detecting malicious Azure sign-ins using heterogeneous graph neural network embeddings rather than brittle rule-based or handcrafted-feature systems [record_id:175]. In the 2024 record, coauthored with Rodrigo Bersa, the focus shifts toward using multi-component large language model systems to automate CVE impact analysis, verification, and VEX justification for container environments [record_id:138].

The strongest collective theme is the use of AI systems to improve cybersecurity triage, detection, and decision support in environments where security data is relational, noisy, and labor-intensive to interpret. The evidence base is narrow—only two records—but both are substantive technical abstracts rather than purely logistical listings. Together they suggest recurring interests in cloud security, automation, machine learning for cyber operations, analyst workflow acceleration, and moving beyond static rules or manual review.

## Research Landscape

The available records consist of two CAMLIS conference talks or presentation abstracts. Both are technical research summaries and appear to describe applied cybersecurity machine learning systems rather than policy, governance, or purely theoretical work. The 2022 record is authored solely by Tadesse Zemichael and discusses malicious Azure Active Directory sign-in detection using heterogeneous relational graph convolutional networks [record_id:175]. The 2024 record is coauthored by Rodrigo Bersa and Tadesse Zemichael and discusses LLM agents for vulnerability identification and verification of CVEs in containerized systems [record_id:138].

The research landscape represented by these records is small but coherent. Both records sit at the intersection of cybersecurity operations and machine learning. The 2022 work belongs primarily to identity and access management, detection engineering, and cloud authentication telemetry. It frames Azure-AD sign-in logs as a rich relational data source involving users, devices, services, authentication mechanisms, and allow/block outcomes [record_id:175]. The 2024 work belongs primarily to software supply chain security and vulnerability management, especially in containerized environments where analysts must decide whether CVEs are actually exploitable or relevant to a given target container [record_id:138].

The two records also show a temporal shift in the type of AI technique being emphasized. The 2022 record focuses on graph neural networks, specifically heterogeneous relational graph convolutional embeddings, as a way to model complex authentication relationships [record_id:175]. The 2024 record reflects the post-LLM-agent research trend, using planner-executor architectures, retrieval-augmented generation, tool use, SBOMs, source code, and documentation to automate vulnerability verification and VEX-style justification [record_id:138]. This does not necessarily indicate a complete shift in research agenda, but it does show Zemichael-associated work tracking major developments in applied AI for cyber defense.

## Major Themes And Trends

A central theme across the records is replacing or augmenting manual, rule-based, or handcrafted security workflows with learned or AI-assisted systems. In the Azure sign-in detection work, the record explicitly contrasts the proposed graph approach with earlier anomalous authentication detection methods that rely on black-box machine learning over handcrafted features or rule-based models [record_id:175]. The stated drawback of rule-based or feature-based systems is that they “fail to generalize for new attacks” and require frequent maintenance, while evolving attacks and connected malicious users are difficult to detect with such methods [record_id:175]. This motivates relational modeling of the authentication environment.

The 2024 vulnerability-management work makes a similar argument in a different domain. It describes vulnerability management in containerized systems as “labor-intensive and time-consuming,” especially when many containers are involved [record_id:138]. Analysts must collect, understand, and synthesize information to decide whether a newly identified CVE requires remediation; if not, they must assign an exemption justification status category using standardized VEX reasoning [record_id:138]. The proposed LLM-based system is intended to automate vulnerability management, verification, and VEX justification, reducing the need for manual CVE verification in target containers [record_id:138].

Another recurring theme is the importance of contextual security reasoning. Neither record treats security events as isolated datapoints. The Azure sign-in detection work models relationships among users, devices, services, authentication nodes, and time-windowed activity, arguing that heterogeneous graphs can capture relations among different entities and learn both structural identity and individual feature identity [record_id:175]. The CVE verification work similarly emphasizes context-sensitive analysis, using a planner to generate a task checklist with up-to-date CVE intelligence and an agent with RAG and tool usage to gather information from container SBOMs, source code, and documentation [record_id:138]. In both cases, the security decision depends on relationships and context rather than on a single static signal.

A third theme is analyst workflow acceleration. The records do not frame AI as merely a detection score generator; they frame it as part of practical cyber operations. In the Azure sign-in work, embeddings of authentication events are fed into a binary classifier or anomaly detection algorithm for scoring purposes, implying use in detection or triage pipelines [record_id:175]. In the vulnerability-management work, the output is a final verdict and VEX justification summary, directly aligned with vulnerability-management decisions and exemption workflows [record_id:138]. This suggests a shared concern with making AI outputs operationally useful.

There is also a trend toward modeling dynamic or evolving security environments. The Azure record discusses evolving attacks, connected malicious users, and the challenge of dynamic sign-in behavior [record_id:175]. Its design separates dynamic “authentication” target nodes from more static user, device, and service nodes, allowing time-changing features to be associated with distinct authentication nodes [record_id:175]. The vulnerability record addresses the changing landscape of CVE intelligence and container composition, using up-to-date CVE intel, SBOMs, code, and documentation to assess whether a vulnerability is actually relevant [record_id:138]. Both records implicitly respond to the problem that security state changes over time and static checks are insufficient.

## Methods, Tools, And Approaches Discussed

The 2022 record presents a heterogeneous graph embedding approach for malicious Azure-AD sign-in detection. It starts from Azure Active Directory sign-in logs, which identify the user, the application or service being accessed, and the target accessed by the identity [record_id:175]. The record formalizes an authentication event as involving a service requested by a user from a device using an authentication mechanism, with the result allowed or blocked [record_id:175]. Rather than extracting handcrafted features from logs, the proposed approach builds a heterogeneous graph to model authentication relationships.

A key method in the 2022 work is relational graph neural network embedding. The record states that the work uses heterogeneous graphs for authentication graph modeling and relational GNN embedding to capture relations among different entities [record_id:175]. To handle node feature sparsity and activity aggregation, aggregation is performed over time windows and node tuples consisting of User, Device, and Service [record_id:175]. The graph separates a target “authentication” node from static nodes such as user, device, and service, so that time-changing features are associated with authentication nodes while static entities remain stable [record_id:175]. A heterogeneous relational graph convolution network, or R-GCN, is then trained to output an embedding for the authentication node, and that embedding is fed into a binary classifier or anomaly detection algorithm for scoring [record_id:175].

The 2024 record presents a multi-component LLM system for vulnerability identification, verification, and VEX justification in containerized environments. The system uses a “Plan-and-Execute-style LLM system” for vulnerability impact analysis [record_id:138]. The workflow begins with an LLM planner that generates a context-sensitive task checklist using up-to-date CVE intelligence [record_id:138]. That checklist is then executed by an LLM agent equipped with retrieval-augmented generation capabilities and tool usage [record_id:138]. Additional LLMs summarize and categorize the gathered information to provide a final verdict [record_id:138].

The inputs in the 2024 system are especially relevant to software supply chain and container security. The record states that the system leverages container Software Bill of Materials, source code, and documentation as input to eliminate the need for manual verification of CVEs in target containers [record_id:138]. It is designed not merely to detect listed vulnerabilities but to determine whether CVEs are false-triggered or require remediation, and to produce final justification summaries comparable to human-labeled justifications [record_id:138]. The reference to VEX reasoning situates the work in standardized vulnerability exploitability communication, where organizations need to document why a vulnerability does or does not affect a product or deployment [record_id:138].

Together, the methods show a preference for architectures that structure AI reasoning around the underlying cyber domain. The graph approach structures detection around relationships among identity entities; the LLM-agent approach structures vulnerability analysis around planner-generated tasks, retrieval, tool use, and evidence synthesis. Neither record suggests a simple, single-model pipeline detached from domain context.

## Notable Talks, Records, And Evidence

The 2022 CAMLIS record, “Heterogenous Graph Embedding for Malicious Azure Sign-in Detection,” is important because it establishes a clear technical contribution by Tadesse Zemichael in identity security and graph-based detection [record_id:175]. The record explains the limitations of rule-based and handcrafted-feature authentication detection, then proposes heterogeneous graph modeling and relational GNN embeddings as an alternative [record_id:175]. Its relevance is strengthened by the use of real-world Azure authentication logs for performance comparison, although the raw record does not provide quantitative results, dataset scale, evaluation metrics, or baselines in detail [record_id:175].

This record is representative of applied machine learning for cloud identity security. It uses Azure-AD sign-in logs and focuses on relations among users, services, devices, authentication mechanisms, and sign-in outcomes [record_id:175]. The proposed model design—separating dynamic authentication nodes from static user/device/service nodes—is a noteworthy contribution because it addresses the temporal nature of authentication events without requiring the entire graph to be modeled as dynamically evolving [record_id:175]. For downstream researchers, this record is likely the best evidence for Zemichael’s work on graph neural networks, malicious sign-in detection, identity telemetry, and Azure security analytics.

The 2024 CAMLIS record, “LLM Agents for Vulnerability Identification and Verification of CVEs,” is important because it shows Zemichael-associated work applying LLM agents to vulnerability management and software supply chain security [record_id:138]. The abstract identifies a concrete operational pain point: analysts manually determining whether CVEs in containerized systems require immediate remediation and, when they do not, assigning VEX exemption justification categories [record_id:138]. The proposed system uses a planner-executor LLM architecture, retrieval-augmented generation, tool use, and multiple LLM components to generate final verdicts and justification summaries [record_id:138].

This 2024 record is representative of the broader movement toward LLM-assisted cyber defense workflows. Its contribution is not simply using an LLM to summarize CVEs; it describes a multi-step agentic system that collects context, checks evidence against container SBOMs, source code, and documentation, and categorizes the outcome according to VEX-style reasoning [record_id:138]. The record also reports experimental results on both synthetic and real-world datasets, claiming high accuracy in capturing false-triggered CVEs and justification summaries on par with human-labeled justifications [record_id:138]. However, as with the 2022 record, the abstract does not include detailed metrics, benchmark design, failure modes, or reproducibility information.

## Gaps, Limits, And Open Questions

The evidence base for this author topic is thin because it consists of only two records. Both are conference abstracts rather than full papers, slide decks, recordings, code repositories, or detailed experimental reports. As a result, the records are useful for identifying themes and technical directions, but they are limited for assessing empirical rigor, reproducibility, deployment maturity, or comparative performance.

For the Azure sign-in detection work, several important questions remain unanswered. The record says performance was compared on real data extracted from real-world Azure authentication logs, but it does not state the size of the dataset, class balance, labeling method, evaluation metrics, baselines, or whether the data included confirmed malicious activity versus simulated or heuristically labeled events [record_id:175]. It also does not explain operational considerations such as false-positive handling, alert triage, model drift, deployment cost, privacy constraints, or how the system performs against novel attack types. The architecture is described clearly at a high level, but downstream researchers would need the full paper or presentation to determine how nodes and edges are constructed, what features are used, how time windows are selected, and how anomaly thresholds are tuned.

For the LLM-agent vulnerability-management work, the main gaps concern evaluation transparency and trustworthiness. The abstract claims high accuracy on synthetic and real-world datasets and says final justification summaries are on par with human-labeled justifications [record_id:138]. However, it does not provide the number of CVEs evaluated, the container types, the definition of “false-triggered CVEs,” the evaluation rubric for VEX justifications, or the rate and severity of incorrect verdicts. It also does not address common LLM-agent risks such as hallucination, stale retrieval, tool errors, prompt injection through documentation or source artifacts, inconsistent reasoning, or adversarial inputs. Because the system is designed to influence remediation decisions, these omissions are significant for assessing operational safety.
