# Topic: Author: Christopher Domas

## Executive Summary

The four records attributed to Christopher Domas describe a focused body of work around unconventional exploit primitives that emerge below, beside, or after the abstractions on which software security normally depends. Across the records, Domas’s talks emphasize that foundational assumptions—hardware failure is rare and fatal, memory addresses map stably to physical storage, compiler output preserves the apparent intent of source code—can be violated in ways that create new privilege-escalation or boundary-bypass opportunities.

The 2025 records center on software-induced catastrophic hardware failure events and Machine Check Exceptions as an attack surface for cross-ring or hardware privilege escalation [record_id:92] [record_id:2010]. The 2026 records broaden the theme: one explores programmable memory-controller or DRAM mapping behavior as a way to break memory isolation without a conventional software bug [record_id:2677], while another examines compiler optimization pipelines as a source of emergent time-of-check/time-of-use vulnerabilities in defensive C patterns [record_id:2683].

Collectively, the records portray Domas as working at the intersection of hardware exploitation, platform security, reverse engineering, exploit development, and the security consequences of abstraction gaps. The evidence is strong for identifying recurring themes and proposed attack classes, but thin on implementation specifics because the records are conference abstracts rather than full papers, slides, code, or transcripts.

## Research Landscape

The record set consists entirely of conference talk or briefing descriptions from Black Hat and DEF CON, spanning 2025 and 2026. Three of the four records are Black Hat USA briefing entries [record_id:92] [record_id:2677] [record_id:2683], and one is a DEF CON 33 talk listing with a YouTube URL and duration metadata [record_id:2010]. The material is therefore presentation-oriented rather than archival research literature: each record summarizes a talk’s premise, novelty claim, and intended demonstration, but does not provide full methodology, artifacts, empirical data, or reproducibility details.

The dominant research area is exploit development and vulnerability discovery, with a strong platform-security and hardware-security emphasis. The two 2025 records describe essentially the same research theme: deliberate synthesis of hardware-failure events from software, manipulation of Machine Check Exception behavior, disruption of privileged CPU operations, and escalation into privileged space [record_id:92] [record_id:2010]. The Black Hat version includes redacted claims about using the vector against an unnamed target and revealing another unnamed hardware vulnerability [record_id:92], while the DEF CON version generalizes this into “all-new hardware vulnerabilities” [record_id:2010].

The 2026 Black Hat records extend the landscape in two directions. “Spaghettifying DRAM” is positioned as hardware/platform exploitation involving DRAM mapping, memory controllers, I/O routing, integrated GPUs, hypervisors, security coprocessors, and bypass of memory boundaries [record_id:2677]. “C and Its Consequences” shifts into reverse engineering and compiler behavior, but retains the same core interest in unexpected exploitability caused by hidden transformations: modern compiler pipelines can rewrite C in ways that introduce TOCTOU vulnerabilities into code that appears defensively written [record_id:2683].

The overall landscape is therefore not a broad survey of all of Domas’s work, but a compact cluster of recent conference abstracts around attacks on assumptions at the boundary between layers: software versus hardware, source versus machine code, virtual memory versus physical DRAM placement, and recoverable computation versus fatal hardware fault.

## Major Themes And Trends

A central theme is the fragility of “impossible” or invariant conditions. The records repeatedly begin from claims that appear foundational: unrecoverable hardware failures should shut systems down [record_id:92] [record_id:2010]; an address should identify a stable memory location, expressed as “&x == &x” [record_id:2677]; and C source logic should not permit a condition such as `x == 1 && x == 2` to become meaningful in the way the abstract suggests [record_id:2683]. Each talk is framed around violating such assumptions and then weaponizing the violation.

Another recurring concern is that security boundaries often rely on subsystems that are treated as passive, deterministic, or trustworthy, but are in fact complex and manipulable. In the machine-check talks, the platform’s response to fatal hardware errors is treated as a security-relevant transition path. The common expectation is that a Machine Check Exception triggers shutdown before damage propagates, but the described research asks what happens if software deliberately creates fatal events, circumvents fail-safes, and keeps the system running in a damaged state [record_id:92] [record_id:2010]. In the DRAM talk, the overlooked component is a final programmable mapping stage beyond CPU privilege, page tables, memory protection checks, IOMMUs, and memory-controller routing [record_id:2677]. In the compiler talk, the hidden subsystem is the multi-stage compiler pipeline—frontend lowering, intermediate representation optimizations, register allocation, and backend code generation—which can reshape source code into behavior not anticipated by developers [record_id:2683].

A third trend is the pursuit of exploit primitives that do not depend on ordinary software bugs. The machine-check records describe “software-only attacks” that synthesize hardware failure events and use them to gain privileged footholds [record_id:92] [record_id:2010]. The DRAM record is explicit that the proposed boundary bypass works “without a single software bug” [record_id:2677]. The compiler record is more nuanced: it discusses an optimizing compiler introducing TOCTOU vulnerabilities “not as a bug, but as a feature” when it transforms a widely used defensive coding pattern [record_id:2683]. In all cases, the novelty lies less in exploiting a specific coding mistake and more in finding security consequences in intended or at least non-buggy behavior of complex systems.

The records also show a movement from destructive instability toward controlled exploitation. The 2025 abstracts are built around catastrophic failure: cosmic ray bit flips, memory degradation, CPU fires, fatal hardware errors, cascading failures, and systems “damaged but alive” [record_id:92] [record_id:2010]. But the claimed contribution is not merely crashing a system; it is learning to inject failure signals during privileged operations, disrupt secure transitions, recover from unrecoverable states, and continue execution with privileged access [record_id:92]. Similarly, the DRAM talk invokes reality-breaking conditions—scrambled data, meaningless pointers, protected memory becoming reachable—but frames them as a reverse-engineered and weaponized exploit primitive [record_id:2677]. The apparent pattern is turning chaos into control.

Finally, the talks frame these findings as both offensive and defensive research opportunities. The DEF CON machine-check record explicitly mentions “machine check research opportunities” for attackers and defenders across technologies and architectures [record_id:2010]. The Black Hat version makes the same point while also referring to a redacted application against an unnamed target and a redacted hardware vulnerability [record_id:92]. The DRAM and compiler records are written more offensively, but their implications are defensive as well: platform designers, compiler developers, systems programmers, and security reviewers must consider whether their trust boundaries survive lower-level remapping or optimization behavior [record_id:2677] [record_id:2683].

## Methods, Tools, And Approaches Discussed

The records do not provide named tools, released code, or step-by-step procedures, but they do describe several methodological approaches.

The 2025 machine-check work appears to involve deliberately generating hardware-failure signals through software, circumventing normal fail-safe behavior, and timing those signals during privileged CPU operations [record_id:92] [record_id:2010]. The described approach includes forcing a system to continue operating after conditions normally considered unrecoverable, observing how disruptions affect secure transitions, and using cascading failure behavior to reach privileged execution contexts [record_id:92]. The Black Hat abstract further claims that the attack can “undo the damage” and recover from the unrecoverable so that the system continues as though nothing happened, now with a privileged foothold [record_id:92]. The DEF CON version repeats the key method but uses less redacted language, saying the vector reveals all-new hardware vulnerabilities [record_id:2010].

The DRAM work centers on reverse engineering and weaponizing a programmable final mapping mechanism that determines where data lands in DRAM [record_id:2677]. The record places this mechanism beyond conventional software-visible layers: CPU abstractions, page tables, privilege levels, memory protection checks, IOMMUs, and memory-controller I/O routing [record_id:2677]. The approach described is to identify that final mapping stage, understand how it can be changed, and use instability in address-to-DRAM mapping to make protected regions of memory reachable across processes, hypervisors, security coprocessors, integrated GPUs, and other platform components [record_id:2677]. The abstract claims this applies to “hundreds of millions of systems,” but does not name the affected platforms or provide evidence in the record itself [record_id:2677].

The compiler-focused work uses reverse engineering of modern compiler pipelines as its primary approach [record_id:2683]. The record describes the compiler not as a direct translator but as a sequence of transformations: frontend lowering, intermediate representation optimization, register allocation, and backend code generation [record_id:2683]. Domas’s stated method is to study complex emergent behavior across those stages and identify a defensive C coding pattern that can be transformed into a TOCTOU vulnerability by the optimizer [record_id:2683]. The notable point is that the vulnerability is framed as a consequence of legal or intentional optimization behavior rather than a compiler defect.

Across all four records, reverse engineering is either explicit or implied. The compiler talk directly says it reverse engineers modern compiler pipelines [record_id:2683]. The DRAM talk directly says it reverse engineers a programmable mapping mechanism [record_id:2677]. The machine-check talks imply experimental platform-level fault research: injecting signals, disrupting CPU transitions, tracing cascading failures, and recovering from supposedly unrecoverable hardware states [record_id:92] [record_id:2010].

## Notable Talks, Records, And Evidence

The most representative 2025 record is the Black Hat USA briefing “Ghosts in the Machine Check - Conjuring Hardware Failures to Breach CPU Privilege Boundaries” [record_id:92]. It gives the clearest statement of the machine-check research arc: unrecoverable hardware errors normally trigger Machine Check Exceptions and shutdown, but the talk proposes software-created fatal events, bypassed fail-safes, targeted disruption during privileged CPU operations, cascading failures, privilege escalation, and post-exploit recovery [record_id:92]. It is also notable for its redactions, which suggest sensitive or not-yet-public target details: the abstract says the vector is used against “[redacted]” to reveal another “[redacted] hardware vulnerability” [record_id:92]. For downstream researchers, this record is the key source for the machine-check privilege-boundary theme, but it also leaves the greatest open questions because important specifics are withheld.

The DEF CON 33 record “Conjuring Hardware Failures for Cross-ring Privilege Escalation” appears to cover the same or closely related work [record_id:2010]. Its raw description is nearly identical to the Black Hat machine-check abstract, including examples of catastrophic failures, software-synthesized fatal events, fail-safe circumvention, disruption of privileged operations, cascading failures, and recovery [record_id:2010]. The title foregrounds “cross-ring privilege escalation,” which helps clarify the intended security impact: movement across CPU privilege rings or analogous privilege boundaries [record_id:2010]. Compared with the Black Hat record, it omits the bracketed redactions and instead says the vector reveals “all-new hardware vulnerabilities” [record_id:2010]. Because it is tied to a YouTube URL and includes a duration tag, it may offer richer evidence if downstream agents can access the talk itself, but the record text here remains only an abstract.

The 2026 Black Hat briefing “Spaghettifying DRAM: Breaking Everything with Memory Collision Exploitation” is the strongest evidence for Domas’s work on memory mapping as an exploit primitive [record_id:2677]. It is conceptually important because it takes a fundamental invariant—an address identifies a variable’s location—and argues that a lower-level programmable DRAM mapping stage can invalidate it [record_id:2677]. The abstract’s scope is broad: processes, hypervisors, security coprocessors, integrated GPUs, and every protected memory region on the platform are described as depending on stable mapping [record_id:2677]. It claims that changing the final mapping makes data scrambled, pointers meaningless, and protected memory reachable, producing a platform-wide boundary bypass without a software bug [record_id:2677]. This is one of the most sweeping claims in the record set, though the abstract does not disclose the actual mechanism, platform family, prerequisites, or limitations.

The 2026 Black Hat briefing “C and Its Consequences: The Source Is Just a Suggestion” is the outlier in that it focuses on compiler transformation rather than hardware-platform behavior, but it fits the same intellectual pattern [record_id:2683]. The record begins with a paradoxical C-like condition and then argues that modern compilers extensively rewrite code through multiple internal stages [record_id:2683]. The claimed contribution is discovering that a widely used defensive coding pattern can be transformed by an optimizing compiler into a TOCTOU vulnerability [record_id:2683]. This record matters because it connects Domas’s broader abstraction-breaking theme to software supply chain and application security: the vulnerability may arise not in the source as written, but in the generated machine code produced by normal compiler behavior.

## Gaps, Limits, And Open Questions

The largest limitation is that all four records are abstracts. They make strong claims about new exploit vectors, vulnerability classes, and platform-wide impact, but they do not include technical details sufficient to validate or reproduce the work.
