# Topic: Author: Mackenzie Jackson

## Executive Summary

The five records attributed to Mackenzie Jackson present a coherent research and teaching profile centered on practical security for modern software delivery: application security, open-source software supply chain defense, cloud and CI/CD security, and the emerging risks of AI agents acting inside development and infrastructure workflows. The records span BSidesLV 2025 and BSidesLV 2026 and include both workshops and talks. They combine hands-on defensive education with operational threat research and exploit-driven analysis.

Across the records, Jackson’s work repeatedly emphasizes that security failures are most consequential when they occur inside trusted automation paths: open-source package ecosystems, CI/CD pipelines, cloud workflows, and developer tooling. In 2025, the focus is strongly on open-source security: one workshop teaches how to build a full-stack security program using open-source tools such as Trivy, GitLeaks, Checkov, ZAP, and OpenGrep [record_id:2440], while another talk describes using large language models to detect vulnerabilities, malicious packages, and APT activity in public software ecosystems, including a claimed interception of a Lazarus Group operation affecting the Ripple cryptocurrency SDK [record_id:2474]. In 2026, the emphasis shifts toward the security implications of AI agents embedded in CI/CD and cloud workflows: one talk describes prompt-injection-driven command execution and secret exfiltration in production systems [record_id:2815], and another frames autonomous AI-driven automation as “synthetic insiders” requiring new identity, permission, ownership, and observability models [record_id:2817].

A separate 2026 hands-on workshop grounds the portfolio in foundational application security education, using vulnerable web apps to teach exploitation and remediation of IDOR, XSS, SSRF, and SQL injection [record_id:2821]. Taken together, the records show Jackson contributing both practitioner-oriented training and forward-looking research on how attackers abuse trust in software and automation systems.

## Research Landscape

The record set consists entirely of conference session descriptions from BSidesLV, covering two years: BSidesLV 2025 and BSidesLV 2026. The formats include workshops and talks. The 2025 records are both attributed solely to Mackenzie Jackson and focus on securing the software supply chain with open-source tools and using LLMs for open-source ecosystem threat detection [record_id:2440] [record_id:2474]. The 2026 records include two solo-authored sessions and one co-authored session with Rein Daelman. These 2026 sessions broaden the focus into AI-agent security in CI/CD pipelines, non-human identities in cloud workflows, and hands-on web application exploitation [record_id:2815] [record_id:2817] [record_id:2821].

The overall research area is best understood as “security at the intersection of software delivery, automation, and trust.” The records repeatedly return to environments where organizations rely on automation and third-party ecosystems: public package registries, GitHub issues, CI/CD systems, cloud automation, internal tooling, and AI-assisted workflows. Jackson’s sessions do not treat these as isolated domains. Instead, they frame them as connected parts of the modern software stack, where a vulnerable package, leaked credential, over-permissioned automation account, or prompt-injected AI agent can become a route to broader compromise.

The talks and workshops also show a strong practical orientation. The “From Code to Cloud” workshop promises live demos of specific tools and integration patterns across CI/CD, IDEs, and pre-commit hooks [record_id:2440]. The “Hacking My First Web App” workshop explicitly rejects checklist-only learning and instead puts participants into a lab of vulnerable applications to practice exploitation and remediation [record_id:2821]. Even the more research-heavy talks emphasize operational examples and reproducible techniques, such as applying frontier LLMs to public package metadata and behavioral signals [record_id:2474], or breaking down exploit chains in AI-enabled CI/CD systems [record_id:2815].

## Major Themes And Trends

### Practical, Budget-Conscious Security Engineering

A recurring theme is that effective security should be operationally practical rather than purely vendor-driven. The 2025 workshop “From Code to Cloud: Securing Your Stack with Open-Source Tools” explicitly contrasts “AI-fueled promises and enterprise price tags” with the needs of teams that “just need tools that work—and won’t wreck the budget” [record_id:2440]. This record frames open-source tooling as a viable foundation for a “full-spectrum security stack” across code and cloud layers.

The workshop’s tool list—Trivy, GitLeaks, Checkov, ZAP, and OpenGrep—indicates coverage across container and dependency scanning, secrets detection, infrastructure-as-code review, dynamic application testing, and code analysis [record_id:2440]. The record also emphasizes workflow integration through CI/CD, IDEs, and pre-commit hooks, suggesting that Jackson’s approach is not only about tool selection but about embedding security checks where developers already work [record_id:2440].

This practical stance is reinforced by the 2026 web application lab, which argues that “Security checklists and ‘Top 10’ lists” are insufficient unless learners understand “how an attacker actually thinks” [record_id:2821]. That workshop focuses on hands-on exploitation of real bug classes and then on fixing them “at the source” [record_id:2821]. Together, these records position Jackson as an advocate for applied security education: tools, labs, workflows, and remediation rather than abstract awareness.

### Open-Source Software Supply Chain As An Attack Surface

The 2025 records strongly foreground open-source software supply chain risk. In the open-source tooling workshop, Jackson identifies attack paths such as Log4Shell, dependency poisoning, and leaked secrets, and frames early detection as a central defensive objective [record_id:2440]. These examples cover several major supply chain failure modes: vulnerable dependencies, malicious dependency manipulation, and credential exposure.

The second 2025 talk, “Inside the Open-Source Kill Chain,” expands this theme into large-scale threat research. It describes a multi-year project using LLMs to analyze public datasets such as changelogs, package metadata, and behavioral signals [record_id:2474]. The record claims this work uncovered “over 900 undisclosed vulnerabilities,” high-severity issues in popular packages such as Axios, and “thousands of malicious packages” in public registries [record_id:2474]. It also states that the research intercepted a live Lazarus Group operation and prevented a backdoor from shipping in the official Ripple XRP cryptocurrency SDK [record_id:2474].

The phrase “open-source kill chain” is significant. The record says the talk maps how attackers abuse trust in public ecosystems “to gain access, deliver payloads, and persist undetected” [record_id:2474]. This suggests Jackson is not merely cataloging individual malicious packages or vulnerabilities but trying to describe attacker lifecycle patterns specific to open-source ecosystems. The supply chain theme therefore evolves from tool-based hardening [record_id:2440] to adversary tracking and kill-chain modeling [record_id:2474].

### LLMs As Both Defensive Amplifiers And Attack Enablers

The records show a notable duality in Jackson’s treatment of AI and LLMs. In 2025, LLMs are presented as a defensive capability for vulnerability discovery and threat triage. “Inside the Open-Source Kill Chain” says that “out-of-the-box frontier LLMs like GPT-4” can augment traditional vulnerability discovery, identify attacker behavior patterns, and assist with triage at scale [record_id:2474]. The evidence described includes analysis of changelogs, package metadata, and behavioral signals, with claimed findings across vulnerabilities, malicious packages, and APT-linked activity [record_id:2474].

By 2026, AI agents are presented as an attack surface in their own right. “Turning GitHub Issues Into RCE” argues that prompt injection is often misclassified as a model safety issue, whereas the real danger appears when AI is “embedded into systems that can act” [record_id:2815]. The talk describes a pattern in which untrusted input enters prompts, model output drives behavior, and that behavior executes with elevated privileges [record_id:2815]. This is a major thematic shift: AI is no longer just an analysis aid but part of the privileged automation fabric.

The “Synthetic Insider” talk extends this idea into cloud identity and access management. It describes AI-driven automation as creating “non-human identities that act autonomously in cloud environments,” operating in CI/CD pipelines and internal tooling with inherited permissions [record_id:2817]. These AI agents are said to have non-deterministic behavior, unclear ownership, and poor auditability [record_id:2817]. In this framing, AI agents become insider-like actors: trusted by systems, capable of action, and potentially influenceable by attackers without traditional credential theft [record_id:2817].

This dual treatment of LLMs is one of the most important trends in the records. Jackson’s work appears to recognize LLMs as useful for defensive scale while simultaneously warning that agentic LLM deployments can introduce new privilege and accountability failures [record_id:2474] [record_id:2815] [record_id:2817].

### Trust Boundaries In CI/CD And Cloud Workflows

Several records converge on the problem of trust boundaries in automation. The open-source tooling workshop covers integration into CI/CD and pre-commit workflows [record_id:2440]. The LLM supply chain talk examines how public package ecosystems can be abused because they are trusted by developers and build systems [record_id:2474]. The AI-agent CI/CD talk makes this trust boundary explicit: GitHub issues and model-generated output can influence privileged pipeline behavior [record_id:2815]. The synthetic insider talk further generalizes the issue to cloud workflows, where autonomous non-human actors interact with APIs and infrastructure using inherited permissions [record_id:2817].

The most concrete example is record 2815. It claims command execution and secret exfiltration across multiple production systems, including Google, Datadog, Vercel, and other Fortune 500 environments [record_id:2815]. In Google’s Gemini CLI, the record says injected instructions caused the agent to call internal tools and write secrets such as `GITHUB_TOKEN`, `GEMINI_API_KEY`, and cloud credentials into public issue data [record_id:2815]. In Vercel workflows, injected payloads in model-generated output were later executed in a shell context, resulting in `GH_TOKEN` exfiltration [record_id:2815]. These examples illustrate a recurring concern: systems designed to automate development work may unintentionally convert untrusted text into privileged action.

The trust-boundary theme is also present in the web application workshop. While less focused on automation, it teaches bug classes such as IDOR, XSS, SSRF, and SQL injection that all involve failures to correctly separate trusted and untrusted inputs or to constrain what an attacker can cause a system to do [record_id:2821]. This gives the broader portfolio continuity: whether the target is a web app, package ecosystem, CI/CD agent, or cloud workflow, the underlying concern is how input, identity, and authority interact.

### Attacker-Centered Learning And Exploit Realism

Another recurring theme is attacker-centered understanding. The web app workshop states that to understand web security, “you have to get your hands dirty breaking code,” and participants will act as attackers to see how bugs work “under the hood” [record_id:2821]. The open-source tooling workshop similarly promises to “unpack real attack paths” including Log4Shell, dependency poisoning, and leaked secrets [record_id:2440]. The CI/CD AI-agent talk breaks down exploit chains and emphasizes why the vulnerability class is difficult to eliminate in practice [record_id:2815].

This attacker-centered approach is not framed as offense for its own sake. The stated endpoint is remediation and prevention: detect and stop attacks early [record_id:2440], assist threat triage at scale [record_id:2474], understand why AI-agent vulnerabilities emerge [record_id:2815], secure non-human identities through ownership, permission scoping, and observability [record_id:2817], and “kill” web bugs at the source [record_id:2821].

## Methods, Tools, And Approaches Discussed

The records describe several concrete methods and approaches.

One major approach is building a layered open-source security stack. The 2025 workshop names Trivy, GitLeaks, Checkov, ZAP, and OpenGrep as tools used to secure “every layer from code to cloud” [record_id:2440]. The associated workflow includes CI/CD integration, IDE integration, and pre-commit hooks [record_id:2440]. This suggests a shift-left and continuous-feedback model: catch dependency risks, secrets, infrastructure misconfigurations, web flaws, and code issues as early as possible in development.

Another approach is LLM-assisted open-source threat detection. Record 2474 describes applying LLMs to public datasets including changelogs, package metadata, and behavioral signals [record_id:2474]. The stated goals are vulnerability discovery, malicious package identification, attacker behavior pattern recognition, and threat triage [record_id:2474]. The record emphasizes “out-of-the-box frontier LLMs like GPT-4” rather than custom-only systems, implying that contemporary general-purpose models can already augment traditional security analysis [record_id:2474]. However, the record does not provide implementation details such as prompts, evaluation criteria, false positive rates, dataset construction, or validation procedures.

The AI-agent CI/CD exploitation work describes prompt injection as a method for influencing privileged automation. The key pattern is: untrusted input enters prompts; model output influences system behavior; the resulting behavior executes with elevated privileges [record_id:2815]. Specific exploit outcomes include command execution and secret exfiltration [record_id:2815]. The record describes attacks against systems where agents interact with internal tools, public issue data, shell contexts, and environment secrets such as GitHub and cloud credentials [record_id:2815].
