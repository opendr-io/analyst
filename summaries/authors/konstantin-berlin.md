# Topic: Author: Konstantin Berlin

## Executive Summary

The two records attributed to Konstantin Berlin are CAMLIS presentations from 2019 and 2023, both centered on practical infrastructure for cybersecurity machine learning at Sophos. Together, they show a consistent focus on the data-engineering and MLOps foundations required to make security ML reliable at production scale: collecting large volumes of telemetry, maintaining high-quality labels, supporting historical reproducibility, joining heterogeneous security data, and keeping infrastructure affordable and maintainable for applied data science teams [record_id:214] [record_id:157].

The 2019 presentation, “Scalable Infrastructure for Malware Labeling and Analysis,” focuses on malware detection ML and the challenge of building large, accurate, historically consistent labeled datasets using AWS-based data lake and warehouse components [record_id:214]. The 2023 presentation, “SQL Driven Infrastructure for Cybersecurity ML Operations,” broadens the scope from malware labeling to cloud-centric cybersecurity detection pipelines, describing a Snowflake-centered, SQL-orchestrated MLOps architecture with immutable external compute components exposed through SQL [record_id:157].

Across both records, Berlin’s distinctive contribution is not the proposal of a new model architecture or detection algorithm, but the articulation of infrastructure patterns that make cybersecurity ML operationally viable: batch-oriented cloud pipelines, data warehouse centricity, scalable storage and joins, historical label management, cost-conscious design, and workflows that reduce dependence on specialized cloud engineering skills [record_id:214] [record_id:157].

## Research Landscape

The available corpus is small but coherent: two CAMLIS talk abstracts, one from 2019 and one from 2023, both authored by Konstantin Berlin and both tied to Sophos or Sophos AI [record_id:214] [record_id:157]. The records are presentation descriptions rather than full papers, code repositories, datasets, or empirical benchmark results. As a result, they provide strong evidence about the topics Berlin presented on and the architectural ideas he emphasized, but limited evidence about implementation details, measured performance, or comparative evaluation.

The research area represented by these records sits at the intersection of cybersecurity ML operations, data engineering, cloud infrastructure, malware analysis, detection engineering, and security data warehousing. The 2019 record is grounded in malware analysis and antimalware model development: it frames the central problem as acquiring more training data and more accurate labels for malware detection models, while preserving the ability to reconstruct historical label states for realistic evaluation [record_id:214]. The 2023 record shifts toward broader cloud-based detection operations, describing a “major paradigm shift” from prevention at edge devices toward cloud-centric detection pipelines over centrally stored estate-wide data [record_id:157].

Both talks are infrastructure-oriented. They emphasize the enabling layer beneath machine learning: data ingestion, batching, replication, storage, SQL joins, scalable compute, labeling workflows, and export paths to training or operational services [record_id:214] [record_id:157]. The records do not describe adversarial ML, malware reverse-engineering methods, feature engineering specifics, or model architectures in depth. Instead, they concentrate on the infrastructure constraints that determine whether ML workflows can be maintained, scaled, audited, and adapted over time.

## Major Themes And Trends

A primary theme across both records is that cybersecurity ML is constrained less by algorithms than by data quality, data volume, and data operations. In the 2019 talk, Berlin states that one of the “best-known secrets” of ML is that more accurate models often come from more training data and more accurate labels, and he explicitly applies this to malware detection models deployed at Sophos [record_id:214]. The 2023 talk echoes this by saying that “the biggest challenge in cybersecurity ML remains data quality,” especially for smaller groups without dedicated engineering operations [record_id:157]. This continuity suggests a sustained view that production security ML depends on robust data infrastructure as much as, or more than, novel modeling.

A second recurring theme is the temporal complexity of security data. The 2019 record emphasizes that malware labels often arrive long after first observation, sometimes months or years later, and that labels from intelligence feeds may be inaccurate, incomplete, or change over time [record_id:214]. This creates a need to join multiple feeds, revise labeling methods across entire datasets, and “roll back” to previous label sets for model development and validation [record_id:214]. While the 2023 record does not focus specifically on historical malware labels, it extends the same concern for data-driven reproducibility and pipeline stability through the idea of immutable infrastructure components that are not removed but autoscaled to zero, so updates do not break existing pipelines [record_id:157].

A third theme is the growing centrality of cloud data platforms in cybersecurity. The 2019 presentation describes an AWS-based approach using regional data lakes, SQS, auto-scaling spot instance clusters, S3 replication, Redshift, and DynamoDB to aggregate, analyze, export, and operationalize malware-related data [record_id:214]. By 2023, the architecture is described as a cloud-based SQL data warehouse approach using Snowflake, with SQL-orchestrated workflows and external Kubernetes or ECS autoscaling clusters accessible through SQL [record_id:157]. This indicates a progression from building scalable cloud infrastructure around AWS data lake and Redshift patterns toward a more explicit “data warehouse first” MLOps model.

A fourth theme is democratization and maintainability of ML infrastructure. The 2023 record argues that SQL-based pipelines are easier to understand and do not require a complex cloud engineering skillset to maintain or modify [record_id:157]. The 2019 record similarly emphasizes that the infrastructure must support a global Data Science team under limited budget constraints, while remaining scalable, agile, and affordable [record_id:214]. In both cases, Berlin’s emphasis is on making sophisticated cybersecurity ML operations feasible for teams that cannot rely on unlimited engineering resources.

A fifth theme is the operational bridge between offline analysis and production security services. The 2019 talk describes exporting data at scale to local storage for ML training and propagating data to edge services, such as replicated DynamoDB, to power a global reputation service [record_id:214]. The 2023 talk describes cloud-centric detection pipelines that combine data across email, firewall, and endpoints, enabling more complex detection logic and orchestrated mitigations or remediations in response to attacks [record_id:157]. Both records therefore connect data infrastructure to real-world security outcomes: antimalware reputation lookups, detection decisions, and remediation workflows.

## Methods, Tools, And Approaches Discussed

The 2019 record presents a concrete AWS-oriented architecture for malware labeling and analysis. It begins from the premise that malware ML requires large, accurate datasets and that security labels arrive from internal and external intelligence feeds with delays and inconsistencies [record_id:214]. To address this, the proposed infrastructure aggregates and batches data across regions into a regional data lake while maintaining GDPR compliance, using SQS, auto-scaling spot instance clusters, and S3 replication [record_id:214]. The data is then batch inserted into a distributed columnar Redshift database for scalable storage, joins, and analysis [record_id:214].

Within Redshift, the 2019 talk emphasizes practical database design issues: organizing tables, defining distribution and sort keys, handling duplicate entries because key uniqueness is not enforced, and performing SQL joins efficiently on a daily cadence [record_id:214]. This is significant because it frames malware labeling as a large-scale relational and temporal data management problem rather than merely a classification problem. The same infrastructure can export data to local storage for ML training or propagate data to edge-facing services, including replicated DynamoDB powering a global reputation service [record_id:214].

The 2023 record describes what Berlin calls an “immutable SQL-driven infrastructure” for cybersecurity MLOps at Sophos AI [record_id:157]. The core idea is to run SQL-orchestrated workflows on top of a cloud-based SQL data warehouse, identified in the abstract as Snowflake [record_id:157]. Non-SQL components are made directly accessible from SQL through external linkage to standard ECS or Kubernetes autoscaling clusters, fronted by a generic batching-first API [record_id:157]. These external components are described as immutable in the operational sense that they are not removed from infrastructure; instead, they are autoscaled to zero, so updates cannot break existing pipelines [record_id:157].

This 2023 approach places SQL at the center of the MLOps workflow. Berlin argues that SQL-written pipelines are easier to understand and reduce the need for complex cloud engineering skills when maintaining or modifying them [record_id:157]. The architecture also assumes that cloud-centralized security data enables richer detections: data from email, firewall, and endpoints can be combined to create more complex detection logic and orchestrate mitigations and remediations [record_id:157]. The method is therefore both technical and organizational: use the data warehouse as the primary orchestration layer, expose specialized compute through stable SQL-accessible interfaces, and reduce operational complexity for ML teams.

Across both talks, batch processing is a notable pattern. The 2019 architecture relies on aggregation, batching, replication, batch insertion, daily SQL joins, and large-scale export [record_id:214]. The 2023 architecture explicitly uses a “batching-first API” for external compute components integrated with SQL workflows [record_id:157]. This suggests that Berlin’s infrastructure thinking favors predictable, scalable batch pipelines for high-volume security data, while still supporting production-facing outputs such as reputation services or cloud detection/remediation pipelines [record_id:214] [record_id:157].

## Notable Talks, Records, And Evidence

The 2019 CAMLIS talk, “Scalable Infrastructure for Malware Labeling and Analysis,” is the earlier and more malware-specific record [record_id:214]. It is notable for its detailed account of why malware labeling is difficult: labels are not usually available at observation time; they may arrive months or years later; they come from multiple internal and external feeds; they may be inaccurate, incomplete, or mutable; and realistic model evaluation may require reconstructing the labels available at a prior time [record_id:214]. These points make the talk a useful source for downstream researchers studying label latency, historical reproducibility, and data lake/warehouse architectures for malware ML.

The same 2019 record is also the most detailed source for specific cloud services and implementation considerations. It references AWS infrastructure, SQS, auto-scaling spot instance clusters, S3 replication, Redshift, distribution and sort keys, duplicate handling, SQL joins, local export for ML training, and replicated DynamoDB for a global reputation lookup [record_id:214]. This makes it the stronger of the two records for understanding concrete implementation concerns in large-scale malware data processing.

The 2023 CAMLIS talk, “SQL Driven Infrastructure for Cybersecurity ML Operations,” is notable because it reframes the problem around a broader industry shift: cybersecurity protection moving from attack prevention on edge devices toward cloud-centric detection pipelines built on centrally stored estate-wide data [record_id:157]. It describes how centralization allows detections to combine observability from multiple points, such as email, firewall, and endpoints, and to support complex mitigations and remediations [record_id:157]. For downstream researchers, this record is important evidence of Berlin’s later emphasis on cloud detection pipelines and MLOps simplification.

The 2023 record is also the clearest articulation of Berlin’s “SQL-driven” or “data warehouse first” approach. It names Snowflake as the cloud SQL data warehouse and describes external non-SQL components linked through ECS or Kubernetes autoscaling clusters behind a generic batching-first API [record_id:157]. It also introduces the idea of immutable external components, retained in the infrastructure but autoscaled to zero to avoid breaking existing pipelines when updates occur [record_id:157]. This record therefore represents a more mature MLOps design philosophy: centralize orchestration in SQL, abstract non-SQL compute behind stable interfaces, and optimize for maintainability.

Together, the two records show an evolution from a 2019 focus on AWS-based malware labeling and analysis infrastructure to a 2023 focus on Snowflake-centered SQL orchestration for broader cybersecurity ML operations [record_id:214] [record_id:157]. The continuity is strong: both talks emphasize scale, cost, batch processing, SQL, data quality, cloud infrastructure, and operational outputs. The shift is in abstraction level and scope: the earlier talk is more concrete and malware-specific, while the later talk is broader, more architectural, and more explicitly positioned as an MLOps strategy [record_id:214] [record_id:157].

## Gaps, Limits, And Open Questions

The evidence base is limited to two talk abstracts. These records establish the topics Berlin presented and summarize architectural claims, but they do not provide full technical diagrams, source code, quantitative performance results, cost figures, incident case studies, or empirical comparisons against alternative architectures [record_id:214] [record_id:157]. Downstream researchers should treat the records as high-level evidence of themes and proposed approaches, not as independently validated proof that the infrastructures achieved specific performance, reliability, or cost outcomes.

Several technical details remain open. The 2019 record says the infrastructure can support the data collection and analysis needs of a global Data Science team while maintaining GDPR compliance and supporting a global reputation lookup, but it does not explain the compliance model, access controls, anonymization strategy, retention policy, or exact cross-region data governance mechanisms [record_id:214]. It also mentions label rollback and historical label records but does not specify the schema design, versioning model, lineage tracking approach, or evaluation protocol used to reconstruct past knowledge states [record_id:214].

The 2023 record describes SQL-orchestrated workflows and immutable external components, but it leaves many implementation questions unanswered. For example, it does not specify how dependency management, schema evolution, feature versioning, testing, monitoring, access control, model deployment, or failure recovery are handled in the Snowflake-centered architecture [record_id:157]. It also states that updates to immutable external components cannot break existing pipelines because old components are not removed, but the abstract does not explain how routing, version pinning, deprecation, security patching, or stale-component risk are managed [record_id:157].
