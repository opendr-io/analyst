## Executive Summary

The two available records attributed to Darryl G. Baker are both BSidesLV 2025 sessions focused on identity-centric security: one is a technical talk on OAuth/OpenID Connect misconfigurations and token replay, and the other is a hands-on class on Active Directory attack and defense. The evidence base is narrow—only two records, both from the same event and year—but the thematic consistency is strong. Baker’s visible work in this dataset is best characterized as identity and access security training with a strong emphasis on misconfiguration, token or credential abuse, enterprise exploitation paths, and actionable defensive hardening.

## Major Themes And Trends

A dominant theme across Baker’s recorded work is that identity systems become security liabilities when they are implemented or configured incorrectly. In the OAuth/OIDC talk, the record states that OAuth and OpenID Connect are “the backbone of modern identity and access management,” but that “poor implementations leave organizations dangerously exposed” [record_id:2383]. The named weaknesses—missing `state` parameters, improperly validated discovery documents, and token validation failures—are not flaws in the abstract idea of federated identity, but in how organizations build or integrate identity flows [record_id:2383]. Similarly, the Active Directory class centers on “common misconfigurations” and real-world attack techniques in AD environments [record_id:2388]. This suggests a recurring concern with the gap between secure design principles and operational reality.

A second theme is token, credential, and delegation abuse. The OAuth/OIDC session focuses explicitly on token replay, including capturing and reusing a JWT for unauthorized access [record_id:2383]. The AD class covers attacks that rely on Windows authentication and delegation behavior, including NTLM Relay, Kerberoasting, Machine Account Quota abuse, and Unconstrained Delegation [record_id:2388]. Although the technologies differ, both sessions focus on the ways attackers exploit trust artifacts—tokens, tickets, delegated privileges, machine accounts, and protocol flows—to impersonate users or expand access.

A third theme is hands-on attacker-defender learning. Baker’s OAuth/OIDC abstract says the session will “move beyond theory” and demonstrate exploitation, while also providing mitigation knowledge and open-source scripts and tools for reproducing attack scenarios in labs [record_id:2383]. The Active Directory class is explicitly “hands-on,” gives each student a lab environment, has participants perform attacks step-by-step, and then implement defensive measures [record_id:2388]. These records collectively frame security education as experiential: participants should not merely learn that a risk exists, but should see how it works, reproduce it, and then apply controls.

A fourth theme is enterprise applicability. The OAuth/OIDC talk promises that attendees will learn to identify, exploit, and mitigate flaws “in enterprise environments” [record_id:2383]. The Active Directory class is directly oriented toward system administrators, IT professionals, and security practitioners responsible for real AD environments [record_id:2388]. The mitigations are operationally concrete: restricting delegation, reducing MachineAccountQuota, disabling unnecessary services, enabling LDAP signing, increasing LDAP diagnostic levels, and configuring Windows Event Forwarding from the domain controller to a log aggregator [record_id:2388]. This practical enterprise framing distinguishes the records from purely academic vulnerability research.

There is also a visible trend toward balanced offensive and defensive instruction. Neither record is solely about attack. The OAuth/OIDC session includes exploitation and mitigation of OAuth/OIDC flaws [record_id:2383]. The AD class pairs each offensive topic with defensive measures and logging practices [record_id:2388]. This dual-use structure is important for downstream researchers: the records should be read as security training material aimed at improving defensive capability through controlled understanding of attack methods.

## Methods, Tools, And Approaches Discussed

The OAuth/OIDC session describes a technical workflow for finding and demonstrating identity protocol weaknesses. The methods include analyzing OAuth and OIDC flows for missing `state` parameters, checking validation of discovery documents, and identifying token validation failures [record_id:2383]. These are presented as exploitable conditions that may enable authentication bypass, user impersonation, or unauthorized access through token replay [record_id:2383]. The record specifically names OWASP ZAP as the tool used to intercept and reuse a captured JWT in a live token replay demonstration [record_id:2383]. It also mentions that attendees will receive open-source scripts and tools to reproduce the scenarios in their own labs, though the record does not identify those scripts by name [record_id:2383].

The Active Directory class uses a dedicated Azure lab environment as its main instructional architecture. Each student is expected to work with three virtual machines: a Windows 10 client, a Windows Server 2019 domain controller, and an Ubuntu VM configured with relevant attack tools [record_id:2388]. The Ubuntu machine includes Docker containers for NTLM relay, indicating a containerized approach to parts of the attack tooling [record_id:2388]. The record names several attack techniques: NTLM Relay, Kerberoasting, Machine Account Quota abuse, and Unconstrained Delegation [record_id:2388]. These methods cover different categories of AD weakness, including authentication relay, service ticket abuse, default or overly permissive machine account creation settings, and unsafe delegation models.

On the defensive side, the AD class emphasizes hardening and monitoring. Defensive measures include restricting delegation, reducing MachineAccountQuota, disabling unnecessary services, and enabling LDAP signing [record_id:2388]. It also covers logging practices such as increasing LDAP diagnostic levels and configuring Windows Event Forwarding from the domain controller to a log aggregator [record_id:2388]. These details show that Baker’s approach includes both preventive controls and detection/visibility improvements.

The methods across both records share a similar educational pattern:

1. Identify a common identity or access-management weakness.
2. Demonstrate how the weakness can be exploited in a controlled environment.
3. Teach practical mitigation or hardening steps.
4. Provide or use lab tooling so practitioners can reproduce the scenario and internalize the lesson.

This pattern is clearest in the AD class because the record explicitly describes step-by-step attacks followed by defensive measures [record_id:2388], but it is also present in the OAuth/OIDC session’s promise of live demonstrations, mitigation guidance, and reproducible lab materials [record_id:2383].
