# Topic: Cloud, infrastructure, and CDR

## Executive Summary

Across 66 records, this topic is dominated by practical security research on modern cloud infrastructure: cloud-provider control planes, identity systems, containers, Kubernetes, managed databases, SaaS platforms, cloud-native telemetry, and cloud detection/response workflows. The strongest cluster of evidence comes from 2025–2026 conference talks and workshops, especially Black Hat, DEF CON, BSidesSF, BSidesLV, CAMLIS, Prompt||GTFO, and [un]prompted. The records collectively show a research area shifting from generic “cloud misconfiguration” toward deeper study of provider-specific trust boundaries, machine identities, managed service internals, hybrid identity paths, runtime workload isolation, and cloud-native detection gaps.

A recurring message is that cloud environments concentrate risk in opaque shared infrastructure and identity-mediated access paths. Azure API Connections may expose connected APIs and secrets across tenants [record_id:15]; Amazon ECS co-tenancy can enable IAM privilege hijacking between containers [record_id:38]; GCP Identity-Aware Proxy misconfigurations and a claimed new IAP vulnerability can allow data exfiltration despite no public IPs and hardened perimeters [record_id:2495]; and hybrid Active Directory-to-Entra ID trust can enable tenant compromise with limited logging [record_id:76]. Multiple records also emphasize that cloud providers’ managed services are not automatically safe: managed Postgres, Kafka, ZTNA/SASE clients, bare-metal clouds, AI infrastructure, and VMware ESXi all appear as major attack surfaces [record_id:2063] [record_id:1993] [record_id:2062] [record_id:1955] [record_id:75] [record_id:40].

Container and Kubernetes security is a major subtheme. Records discuss Kubernetes audit-log anomaly detection using graph neural network embeddings [record_id:153], secure Kubernetes debugging [record_id:2218], policy-based deployment prevention with Kyverno and runtime enforcement with KubeArmor [record_id:2429], cluster disruption from common Kubernetes configurations [record_id:2446], seccomp profile generation and deployment for containers [record_id:2456], and container vulnerability verification using LLM agents and SBOM/VEX workflows [record_id:138]. These records frame Kubernetes as both an operational substrate and a security challenge where prevention, telemetry, runtime controls, and identity design must be combined.

Detection and response appears less as a single CNAPP/CDR product category and more as a set of operational capabilities: centralized cloud telemetry, cloud audit logs, behavioral and atomic indicators, cloud attack-path mapping, security data lakes, scalable MLOps pipelines, and incident scoping. Examples include Sophos’s SQL-driven cloud MLOps/detection infrastructure built on Snowflake and autoscaling ECS/Kubernetes components [record_id:157], AWS-based malware labeling infrastructure using SQS, spot instances, S3 replication, Redshift, and DynamoDB [record_id:214], Luminaut for discovering internet-exposed cloud resources and building audit-log timelines [record_id:2414], and cloud-specific IOC work covering IAM metadata, container IDs, and API activity [record_id:2281]. At the same time, several talks argue that detection is constrained by logging gaps or speed: Google Workspace attacks can bypass monitoring and require investigation “even without sufficient logs” [record_id:2312], Entra hybrid abuse may leave few useful logs [record_id:76], and AI-assisted cloud attacks may compress known techniques into timelines that strain traditional detection models [record_id:2355].

The research landscape also shows cloud being used as attacker infrastructure. Records cover abuse of IPFS, Codex, NFTs, and Cloudflare R2 for resilient command-and-control and phishing hosting [record_id:1896], Google Calendar and other Google services for RAT and SOCKS tunneling [record_id:2097], cloud providers and storage systems in an operational relay box network [record_id:2025], and abandoned AWS S3 buckets as a hypothetical supply-chain worm source [record_id:1972] [record_id:2134]. These records broaden “cloud security” beyond defending cloud workloads to include detecting abuse of legitimate cloud services and distributed storage as adversary infrastructure.

Overall, the evidence is strong that cloud and infrastructure security research is converging around identity, workload isolation, managed-service internals, telemetry quality, and multi-cloud operationalization. Evidence is thinner on comparative effectiveness of CNAPP/CDR commercial platforms, longitudinal measurements of mitigation adoption, and standardized cloud telemetry schemas. Many records are abstracts rather than full papers, so downstream researchers should treat claims about undisclosed vulnerabilities, “first-ever” RCEs, and industry-wide susceptibility as leads requiring validation against full talks, advisories, patches, and vendor responses.

## Research Landscape

The records are mostly conference abstracts, workshop descriptions, and talk summaries rather than formal peer-reviewed papers. Black Hat and DEF CON records tend to emphasize exploit research against specific cloud services or infrastructure components, often with claims of undisclosed vulnerabilities or live demonstrations. Examples include Azure API Connections leading to cross-tenant Key Vault secrets leakage [record_id:15], Amazon ECS IAM privilege hijacking [record_id:38], NVIDIA Container Toolkit container escape affecting AI cloud infrastructure [record_id:75], VMware ESXi VM escapes and failed patch analysis [record_id:40], bare-metal cloud compromise paths [record_id:1955], ZTNA/SASE platform weaknesses [record_id:2062], Kafka server RCE [record_id:1993], and cloud-hosted Postgres RCE [record_id:2063] [record_id:2528].

BSides records are more operational and practitioner-oriented. BSidesSF and BSidesLV include talks on hybrid egress control [record_id:2261], cloud vulnerability remediation [record_id:2309], shadow cloud discovery [record_id:2414], Kubernetes prevention controls [record_id:2429], Kubernetes cluster “shenanigans” [record_id:2446], seccomp for container hardening [record_id:2456], multi-cloud red/blue training [record_id:2489] [record_id:2490] [record_id:2491] [record_id:2492], workload identity and secretless authentication [record_id:2396], machine identity attack paths [record_id:2484], and CSP-managed non-human identities [record_id:2552]. These records indicate a community focus on implementable controls, training labs, open-source tooling, and engineering workflows.

CAMLIS records bring a machine-learning and infrastructure-operations lens. They include container vulnerability analysis agents and alert triage agents improved through test-time compute [record_id:117], Relational Inference GraphRAG for cloud configuration knowledge graphs [record_id:118], LLM agents for CVE verification and VEX justification in containerized systems [record_id:138], GNN embeddings for Kubernetes audit-log threat detection [record_id:153], SQL-driven cloud MLOps infrastructure for security detection pipelines [record_id:157], AWS infrastructure for scalable malware labeling and analysis [record_id:214], and serverless AWS binary-analysis pipelines [record_id:235]. These records are especially relevant to cloud detection and response because they focus on how large-scale telemetry and analysis infrastructure is built.

Prompt||GTFO and [un]prompted records show the intersection of cloud security with AI systems. AI agents are described as vulnerable to IDOR, sandbox escape, cloud metadata access, and SSRF through tool parameters [record_id:2209]. A secure Kubernetes debugging tool is described as being built with prompt engineering while following Kubernetes security principles [record_id:2218]. Confidential AI records describe TEEs, remote attestation, and protection of prompts, context, and models in public and hybrid cloud environments [record_id:2343]. Another record describes AI-assisted attacks compressing an AWS escalation from stolen credentials to admin in eight minutes [record_id:2355]. These records suggest cloud security teams increasingly need to understand AI workloads, AI agents, and AI-assisted adversary operations.

There are also records only secondarily tied to cloud and infrastructure, but they still illuminate adjacent concerns. Windows Server Failover Cluster abuse points to high-availability infrastructure as an overlooked enterprise attack surface [record_id:10]. Cellular IoT compromise can become a route to backend cloud and private network attacks [record_id:44]. Internet-scale scanning can map systemic risk concentration across vendors and sectors [record_id:66]. Solar microgrid vulnerabilities include unauthenticated cloud APIs and shared cloud infrastructure across cloned OEMs [record_id:1978]. Third-party risk, product security, SaaS attack-surface mapping, confidential computing, and post-quantum cloud monitoring also appear as adjacent but relevant concerns [record_id:2481] [record_id:2384] [record_id:2298] [record_id:2294] [record_id:1996].

## Major Themes And Trends

### 1. Identity and trust boundaries are central cloud attack surfaces

Many records treat identity—not network reachability—as the decisive cloud security boundary. Azure API Connections are described as integration backbones for Logic Apps, Power Apps, and Power Automate that can be exploited for near-unrestricted access to connected APIs, including cross-tenant Key Vault secret leakage [record_id:15]. Microsoft Entra ID appears in multiple records: one focuses on OAuth authentication/authorization confusion enabling access to internal Microsoft applications [record_id:20], while another describes advanced lateral movement from on-premises Active Directory to Entra ID through undocumented trust flows that may bypass authentication and MFA and leave few logs [record_id:76].

Machine and workload identity are recurring concerns. BSidesLV records call for rethinking long-lived application and workload credentials [record_id:2468], demonstrate Kubernetes clusters in AWS and Azure authenticating to each other with no manually managed secrets through AWS IRSA and Azure Workload ID [record_id:2396], and examine risks in CSP-managed non-human identities across major clouds [record_id:2552]. Another talk frames multi-cloud misconfiguration and machine identity as attack-path problems, recommending tools such as SecBridge, Cartography, and AWSPX to map IAM permissions and visualize attack paths [record_id:2484]. Hidden access paths and engineer-created backdoors to “crown jewels” are also discussed as an access-behavior and remediation problem [record_id:2249].

The trend is toward treating cloud identity as dynamic infrastructure: service accounts, managed identities, OAuth consent, workload federation, API connectors, and hidden access relationships. The evidence is strong across multiple clouds, but the details are highly provider-specific.

### 2. Managed services and shared infrastructure create unexpected privilege and isolation failures

A major pattern is that managed and shared infrastructure can collapse assumptions about tenant isolation or privilege separation. Amazon ECS is described as vulnerable to ECS-cape, where a low-privileged task on an ECS instance can hijack IAM privileges of higher-privileged containers on the same EC2 machine through an undocumented ECS protocol [record_id:38]. NVIDIA Container Toolkit is described as a foundational AI managed-infrastructure component whose vulnerability enables container escape to the host and often Kubernetes cluster compromise; the authors claim major AI platforms were susceptible, with some cases of unauthorized cross-tenant data access [record_id:75]. Bare-metal cloud environments are described as vulnerable to legacy firmware, hardware, provisioning, and network attacks that can lead to multi-tenant compromise [record_id:1955].

Managed data and messaging services also appear as targets. Kafka server-side RCE is presented as a first-ever Kafka Server/Broker RCE, with related ecosystem vulnerabilities and potential impact on cloud service providers [record_id:1993]. Postgres RCE in cloud databases via CVE-2024-10979 appears in both DEF CON and BSidesLV records [record_id:2063] [record_id:2528]. VMware ESXi VM escape research emphasizes that a failed patch left vulnerabilities in VMware hypervisor products for two years and discusses ESXi sandbox bypass and kernel privilege escalation [record_id:40]. Cloud-based VPN/ZTNA/SASE platforms are framed as inheriting legacy VPN weaknesses while adding cloud attack surfaces, including authentication and posture-check bypass, IPC abuse, local privilege escalation, traffic-steering circumvention, and malicious ZTNA servers [record_id:2062].

These records collectively suggest that “managed” does not mean “risk-free.” The attack surface often moves into opaque provider protocols, client components, provisioning workflows, virtualization layers, and co-tenancy decisions.

### 3. Kubernetes and containers are moving from visibility-only toward prevention and runtime containment

Kubernetes security records show a progression from observing and detecting cluster behavior to preventing unsafe workloads and constraining runtime behavior. Kubernetes audit logs are described as rich telemetry for detecting attacks, and one CAMLIS record models sequences of dependent Kubernetes session events as graphs for graph classification and anomaly detection [record_id:153]. BSidesLV’s “Detect and Respond? Cool Story” explicitly argues against relying only on post-deployment detection, showing Kyverno to block dangerous workloads before deployment and KubeArmor to enforce runtime behavior [record_id:2429]. Seccomp is presented as an underused but effective syscall-filtering defense, with practical profile generation and a live Apache Druid exploit containment case study using tools such as Kubescape [record_id:2456].

Other Kubernetes records are operational. Dallas Debugger is described as a secure Kubernetes debugging tool that lets developers debug pods, run scripts, check network connectivity, and query APIs without compromising cluster security, following CIS benchmarks and using in-memory operations with no disk writes [record_id:2218]. Kubernetes “cluster shenanigans” research began as interview-question brainstorming and found that common configurations can lead to surprising cluster disruptions [record_id:2446]. Secretless authentication demonstrations use two Kubernetes clusters, one in AWS and one in Azure, with Terraform source code and workload identity primitives [record_id:2396]. Secret scanning at scale uses Kubernetes and cloud credits for distributed scanning of VS Code extensions and public sources [record_id:2408].

The trend is toward layered Kubernetes defense: admission control, runtime policy, syscall filtering, safer debugging, workload identity, audit-log analytics, and secure-by-design cluster operations.

### 4. Cloud detection and response depends on telemetry quality, context enrichment, and scalable data infrastructure

Several records address the plumbing required for cloud detection and response. Sophos’s SQL-driven infrastructure record describes a shift from edge-device prevention to cloud-centric detection pipelines over centrally stored data from a customer estate, enabling correlation across email, firewall, and endpoints and orchestration of mitigations [record_id:157]. The proposed architecture uses SQL workflows on a cloud-based data warehouse, Snowflake, with non-SQL components accessible through ECS/Kubernetes autoscaling clusters and immutable external components [record_id:157]. Earlier Sophos work used AWS SQS, autoscaling spot instance clusters, S3 replication, Redshift, and DynamoDB to build scalable malware labeling and analysis infrastructure with historical label rollback and GDPR compliance [record_id:214]. CarbonBlack’s serverless AWS file-processing pipeline for binary analysis similarly emphasizes scalable ingestion, preprocessing, serverless Docker containers, storage, and modular feature extraction [record_id:235].

Cloud-specific detection content appears in records on Kubernetes audit logs [record_id:153], cloud IOCs such as IAM metadata, container IDs, and API activity [record_id:2281], Google Workspace logging gaps [record_id:2312], Luminaut’s cloud exposure discovery and audit-log timeline layering [record_id:2414], and multi-cloud offensive/defensive labs for AWS, Azure, and GCP [record_id:2489] [record_id:2490] [record_id:2491] [record_id:2492]. AI-first SOC discussions appear in a BSidesSF panel on securing cloud-native AI systems and building modern SOCs [record_id:2259], while CAMLIS work evaluates test-time compute for a container vulnerability analysis agent and a server alert triage agent [record_id:117].

However, the records repeatedly warn that detection can be undermined by missing telemetry. Hybrid Entra techniques may leave few useful logs [record_id:76]. Google Workspace attackers can exploit logging gaps to escalate privileges, exfiltrate data, and evade detection [record_id:2312]. AI-assisted campaigns may compress known AWS escalation techniques into speeds that “break traditional detection models” [record_id:2355]. This makes telemetry design and response automation central open problems.

### 5. Cloud services are also adversary infrastructure

Multiple records focus less on compromising cloud environments and more on abusing legitimate cloud or distributed services for attacker operations. Distributed replication systems such as NFTs, IPFS, Codex, and Cloudflare R2 are described as resilient C2 infrastructure, payload delivery systems, and phishing hosting that complicate takedowns [record_id:1896]. Google Calendar RAT and a new SOCKS tunneling-over-Google-services tool show how trusted cloud platforms can provide covert C2 and traffic forwarding that evades traditional monitoring [record_id:2097]. A Chinese operational relay box network record is aimed partly at cloud providers trying to stop abuse and claims to name cloud providers, data storage systems, tools, domains, emails, and passwords used to run the network [record_id:2025].

Abandoned or mismanaged cloud storage is another attacker-infrastructure vector. A supply-chain talk describes a hypothetical crypto worm sourced from compromised abandoned AWS S3 buckets, with 5,155 buckets affected before the scenario was prevented by claiming and taking down buckets [record_id:1972] [record_id:2134]. Solar supply-chain research describes insecure firmware, exposed APIs, and rebadged devices using shared cloud infrastructure, allowing vulnerabilities to propagate across cloned OEMs [record_id:1978]. Pig-butchering and money-laundering infrastructure tied to FUNNULL CDN-hosted campaigns also appears as an example of cloud/CDN-enabled criminal infrastructure [record_id:2301].

These records highlight a defender challenge: cloud abuse often hides inside legitimate platforms, storage services, CDNs, and distributed systems, requiring provider cooperation, abuse detection, and new forms of telemetry.

### 6. Risk management is becoming empirical, cloud-specific, and tied to attack paths

Several records position cloud security as a governance and risk-management problem. Internet-scale scanning is proposed as a way to build empirical maps of systemic cyber risk and concentration risk across vendors, sectors, technologies, and digital supply chains [record_id:66]. Third-party vendor cyber risk is framed as increasingly important as organizations adopt cloud and AI, expanding the attack surface and introducing ecosystem vulnerabilities [record_id:2481]. SaaS attack-surface mapping is presented as a concept with current techniques, novel approaches, and an open-source tool release [record_id:2298].

Cloud vulnerability management is also reframed. One BSidesSF talk asks whether traditional vulnerability management is dead and proposes a cloud risk remediation program involving risk prioritization, triage, IaC, and cloud-native mitigating controls [record_id:2309]. Shadow cloud deployments are described as weak links because forgotten proof-of-concept or test environments may be unmonitored, undocumented, and outside standard controls; Luminaut is presented as a tool to scope exposed services, resources, and timelines [record_id:2414]. Product security records distinguish application, platform, and product security in a talk by a Google Cloud security engineer [record_id:2384]. AI resilience records add disaster recovery, availability, foundational security, audit/compliance, and privacy requirements for AI chat/search products [record_id:2257].

The records show a move from checklist-based cloud security toward empirical exposure measurement, attack-path modeling, risk prioritization, and operational remediation.

### 7. Confidential computing and AI infrastructure are emerging cloud security frontiers

AI and confidential computing appear throughout the topic. NVIDIA Container Toolkit vulnerabilities are presented as a path to compromising AI providers and Kubernetes clusters [record_id:75]. Bare-metal clouds are tied to high-performance machine-learning workloads and secure ML workflow compromise [record_id:1955]. Confidential computing is described as isolating data and cryptographically proving what code ran in cloud environments [record_id:2294], while Confidential AI extends hardware-based TEEs and remote attestation to protect prompts, context, inference data, and models in public and hybrid clouds [record_id:2343].

AI agent security also intersects with cloud infrastructure. A Prompt||GTFO record reports compromising 7 of 16 YC Spring batch AI agents, including code sandbox subversion leading to arbitrary code execution and cloud metadata access, plus SSRF leaking git credentials [record_id:2209]. Agentic cloud infrastructure work uses GraphRAG to transform cloud configuration into a security-enriched knowledge graph for natural-language reasoning over deployed infrastructure [record_id:118]. AI-assisted attack campaigns are described as using known techniques at greater speed and scale, including an eight-minute AWS escalation from stolen credentials to admin [record_id:2355].

The evidence suggests that cloud security teams must assess AI platforms not only for model risks but also for classic cloud risks: metadata access, sandbox escape, workload isolation, credentials, attestation, and telemetry.

## Methods, Tools, And Approaches Discussed

The records describe a wide range of operational and research methods.

For cloud detection and analytics, several records focus on graph and data-warehouse approaches. Kubernetes audit sessions are modeled as graphs, classified with GNN embeddings, and then used downstream for anomaly detection [record_id:153]. RIG-RAG transforms cloud configuration data into a security-enriched knowledge graph to support natural-language reasoning about infrastructure [record_id:118]. Sophos’s immutable SQL-driven infrastructure uses SQL-orchestrated workflows on Snowflake, with external ECS/Kubernetes autoscaling components exposed through a batching-first API [record_id:157]. AWS-based malware-labeling infrastructure uses regional data lakes, SQS, autoscaling spot instances, S3 replication, Redshift table design, and DynamoDB edge propagation [record_id:214]. CarbonBlack’s binary-analysis architecture uses AWS serverless design patterns, serverless Docker containers, and modular feature extraction [record_id:235].

For Kubernetes and container hardening, the tools and controls include Kyverno for blocking unsafe workloads before deployment, KubeArmor for runtime behavior enforcement [record_id:2429], seccomp for syscall filtering with profile generation and Kubescape [record_id:2456], secure Kubernetes debugging through Dallas Debugger [record_id:2218], and Kubernetes audit-log analytics [record_id:153]. Cluster security research also emphasizes testing common configurations for disruption potential [record_id:2446]. The NVIDIA Container Toolkit case underscores that container runtime/tooling vulnerabilities can defeat Kubernetes isolation [record_id:75].

For identity and access management, records discuss Azure Resource Management, API Management, Custom Connectors, consent servers, and token stores in Azure API Connections [record_id:15]; OAuth authentication/authorization checks in Microsoft Entra ID [record_id:20]; undocumented hybrid AD/Entra authentication flows [record_id:76]; AWS IRSA and Azure Workload ID for secretless Kubernetes cross-cloud authentication [record_id:2396]; CSP-managed non-human identities [record_id:2552]; and attack-path visualization using SecBridge, Cartography, and AWSPX [record_id:2484]. Service mesh security is represented by Yelp’s shift of authentication and authorization from infrastructure to the application layer [record_id:2281].

For exposure management and cloud risk discovery, Luminaut scans AWS and Google Cloud environments to identify internet-exposed services, IPs, compute and networking resources, and timelines from cloud audit logs plus external scanners [record_id:2414]. SaaS attack-surface mapping includes an open-source tool release [record_id:2298]. Internet-scale scanning is proposed for systemic risk visualization across vendors and sectors [record_id:66]. Google Artifact Registry scanning covers vulnerabilities and secrets in Google-owned container images [record_id:2467]. Secret scanning at scale uses Kubernetes, cloud credits, distributed systems, and automation [record_id:2408]. Open-source “code to cloud” workflows include Trivy, GitLeaks, Checkov, ZAP, and OpenGrep integrated into CI/CD, IDEs, and pre-commit hooks [record_id:2440].

For supply-chain and dependency controls, records describe Nix/NixOS for self-hosting packages, SBOM traceability, avoiding vendor lock-in, and reducing Docker image bitrot [record_id:1954] [record_id:2130]. Anthropic’s Dependant enforces dependency posture at the network level across heterogeneous environments [record_id:2308]. LLM agents for CVE triage use container SBOMs, source code, documentation, up-to-date CVE intelligence, RAG, tool use, and VEX justification generation [record_id:138].

For offensive cloud research, methods include exploiting undocumented ECS protocols [record_id:38], manipulating Azure API connections and token stores [record_id:15], abusing AD-to-Entra trust flows [record_id:76], hijacking stateless tunnels such as GRE and VxLAN and manipulating routing protocols like BGP/OSPF [record_id:1907], exploiting ZTNA authentication/device posture checks and IPC [record_id:2062], analyzing provisioning, firmware, and network weaknesses in bare-metal clouds [record_id:1955], exploiting CVE-2024-10979 in cloud-hosted Postgres [record_id:2063] [record_id:2528], and studying Kafka broker/server RCE and ecosystem components [record_id:1993].

For confidential computing and AI infrastructure, methods include TEEs, remote attestation, encrypted computation, and cryptographic proof of executed code [record_id:2294] [record_id:2343]. AI security operations also include test-time compute for cybersecurity agents [record_id:117] and behavioral methodology for attributing AI assistance in attacks when direct proof is unavailable [record_id:2355].

## Notable Talks, Records, And Evidence

Several records are especially representative of the topic’s direction.

**Azure’s Weakest Link? How API Connections Spill Secrets** is central because it illustrates cloud control-plane complexity, low-privilege exploitation, cross-tenant implications, and hidden integration infrastructure across ARM, APIM, Custom Connectors, consent servers, and token stores [record_id:15]. It is a strong example of modern cloud exploitation focused on provider-specific abstractions rather than traditional network exposure.

**ECS-cape – Hijacking IAM Privileges in Amazon ECS** is representative of container co-tenancy risk in cloud-managed orchestration. Its key claim is that a low-privileged task can hijack IAM privileges of higher-privileged containers on the same EC2 instance by exploiting an undocumented ECS protocol [record_id:38]. The record also provides defensive framing: avoid co-locating high- and low-privilege workloads on the same ECS instance [record_id:38].

**Breaking Out of The AI Cage** is notable because it ties container escape, Kubernetes compromise, AI managed infrastructure, cross-tenant data access, and industry-wide cloud-provider impact into one case study [record_id:75]. It is one of the clearest examples of AI infrastructure being fundamentally cloud/container infrastructure.

**Advanced Active Directory to Entra ID Lateral Movement Techniques** is important for hybrid identity. It states that hardening has reduced some classical AD-to-cloud techniques but that new design-level lateral movement paths can bypass authentication and MFA, exfiltrate data, and leave few useful logs [record_id:76]. This record is highly relevant to CDR gaps because it explicitly discusses logging insufficiency.

**SQL Driven Infrastructure for Cybersecurity ML Operations** is one of the strongest CDR infrastructure records. It describes the broader shift from edge-device prevention to cloud-centric detection pipelines over centralized data, and it gives a concrete Snowflake plus ECS/Kubernetes architecture for scalable security MLOps [record_id:157]. Alongside AWS malware-labeling infrastructure [record_id:214] and serverless binary analysis [record_id:235], it shows how detection capabilities depend on scalable cloud data engineering.

**Threat Detection on Kubernetes Logs Using GNN Embeddings** is a strong example of ML applied to cloud-native telemetry. It models Kubernetes audit sessions as graphs and uses embeddings for anomaly detection, tested through adversarial simulation [record_id:153]. This is one of the more methodologically specific detection records.

**Casting Light on Shadow Cloud Deployments** is notable because it addresses a common operational failure mode: forgotten or undocumented environments outside standard controls. Luminaut’s combination of internal cloud scanning, audit-log timelines, and external scanner context is a practical CDR/incident-scoping approach [record_id:2414].

**No IP, No Problem: Exfiltrating Data Behind IAP** is a representative GCP trust-boundary record. It challenges assumptions that no public IPs, VPC Service Controls, and hardened perimeters are sufficient if IAP trust boundaries, IAM bindings, user-supplied headers, or endpoints are misconfigured [record_id:2495].

**Detect and Respond? Cool Story — or Just Don’t Let the Bad Stuff Start** is a concise expression of the prevention trend in Kubernetes security. It argues for blocking unsafe workloads with Kyverno and enforcing runtime behavior with KubeArmor rather than relying only on after-the-fact detection [record_id:2429].

**Hardening Containers with Seccomp** provides a practical runtime-containment method with real exploit context, emphasizing profile generation, tuning, deployment at scale, and pitfalls [record_id:2456].

**Blank Space: Filling the Gaps in Atomic and Behavioral Cloud-Specific IOCs**, embedded in a noisy BSidesSF schedule record, is important because it explicitly names cloud-specific atomic IOCs such as IAM metadata and container IDs and behavioral IOCs such as API activity for cloud detection, hunting, and response [record_id:2281]. The same raw record also contains a service mesh talk about shifting authentication and authorization to the application layer [record_id:2281], making it a useful but noisy evidence source.

**Metal-as-a-Disservice** stands out for bare-metal cloud risk. It argues that high-performance ML workloads on bare-metal providers can be exposed to legacy firmware, hardware, provisioning, and network attacks, including persistent firmware implants and sensitive network interception [record_id:1955].

**Rusty pearls: Postgres RCE on cloud databases** appears in both DEF CON and BSidesLV forms and is notable as a managed-database vulnerability case centered on CVE-2024-10979 [record_id:2063] [record_id:2528].

**The multi-cloud BSidesLV workshop series** is important evidence of training demand for AWS, Azure, and GCP offensive/defensive operations, APT-style TTP simulation, and blue-team defense in enterprise cloud infrastructure [record_id:2489] [record_id:2490] [record_id:2491] [record_id:2492].

**Confidential Computing** and **Confidential AI** records represent the defensive frontier of cloud data protection. They focus on isolation, cryptographic proof of executed code, TEEs, and remote attestation for protecting customer data, inference data, prompts, context, and models in public/hybrid clouds [record_id:2294] [record_id:2343].

## Gaps, Limits, And Open Questions

The records provide strong evidence of active research interest, but several limitations remain.

First, many records are talk abstracts making bold claims about undisclosed or newly discovered vulnerabilities. Claims such as cross-tenant Azure Key Vault leakage [record_id:15], undocumented ECS privilege hijacking [record_id:38], all-major-AI-platform susceptibility to NVIDIA Container Toolkit escape [record_id:75], first-ever Kafka Server RCE [record_id:1993], and a new GCP IAP exfiltration vulnerability [record_id:2495] require validation through full presentations, CVEs/advisories, reproduction details, vendor statements, and patch status.

Second, the records do not provide a systematic comparison of CNAPP, CSPM, CWPP, CIEM, or CDR products. They discuss many capabilities associated with those categories—cloud exposure scanning, IAM attack paths, audit logs, workload controls, vulnerability remediation, and detection engineering—but not market-level evaluations or controlled benchmarks. Downstream researchers should not infer tool effectiveness beyond what specific records claim.

Third, cloud telemetry remains an unresolved problem. Several records identify logging gaps or low-quality signals, especially in Entra hybrid flows [record_id:76] and Google Workspace [record_id:2312], while others propose richer indicators and graph-based analytics [record_id:153] [record_id:2281]. The records do not establish a standard minimum telemetry set for cloud detection and response, nor do they provide comparative detection coverage across AWS, Azure, and GCP.

Fourth, prevention versus detection is an explicit tension but not resolved. Kubernetes records advocate prevention through Kyverno, KubeArmor, and seccomp [record_id:2429] [record_id:2456], while detection records emphasize audit logs, GNN embeddings, IOC enrichment, and incident scoping [record_id:153] [record_id:2281] [record_id:2414]. The evidence does not quantify tradeoffs such as false positives, developer friction, operational overhead, or incident reduction.

Fifth, AI’s impact on cloud security is visible but still immature. Records discuss AI-assisted cloud operations [record_id:118], AI-based vulnerability management [record_id:138], test-time compute for agents [record_id:117], AI agent compromise leading to metadata access [record_id:2209], confidential AI [record_id:2343], and AI-assisted attacks speeding AWS escalation [record_id:2355]. What remains unclear is how widely these patterns occur in production, how to distinguish AI-assisted attacks reliably, and how cloud SOCs should adapt.

Sixth, systemic and third-party cloud risk is recognized but unevenly evidenced. Internet-scale scanning for concentration risk [record_id:66], third-party vendor risk [record_id:2481], SaaS attack-surface mapping [record_id:2298], and cloud/CDN-enabled scam infrastructure [record_id:2301] point to ecosystem-level risk. But the records do not provide a unified model for quantifying dependency concentration, shared cloud infrastructure exposure, or cascading SaaS failures.

## Coverage Addendum

The omitted BSidesSF record adds a concise but important framing to the cloud detection-and-response discussion: attackers are extending “living off the land” tradecraft into “living off the cloud” tradecraft. Matan Mittelman’s talk, **“From LOL to LOC: LOLBins Are No Laughing Matter,”** explicitly argues that traditional LOLBins-style abuse is evolving because “cloud services [are] becoming a core part of engineering,” making them “a high-value attack vector” for adversaries [record_id:2273]. This strengthens the existing report’s theme that cloud services are not only assets to defend, but also legitimate platforms and workflows that attackers can repurpose for stealth, persistence, movement, or operational leverage.

This record is especially relevant to the report’s sections on cloud-specific IOCs, cloud abuse, and CDR telemetry. While other records discuss concrete abuse of Google services for C2 and SOCKS tunneling, distributed storage and Cloudflare R2 for resilient C2 and phishing, and cloud-specific atomic/behavioral IOCs, [record_id:2273] supplies the conceptual bridge: defenders should expect adversaries to use normal cloud-native services and engineering workflows in ways analogous to LOLBins on endpoints. In CDR terms, this implies that detection cannot rely only on known malicious binaries, public IP exposure, or obvious exploit events; it must model legitimate cloud service usage, engineering behavior, API activity, identity context, and deviations from expected workflows [record_id:2273].

The evidence in this record is brief—a schedule abstract rather than a technical paper or full talk transcript—so it does not provide specific LOC techniques, tool names, telemetry fields, or detection logic. Its strength is thematic rather than evidentiary: it reinforces an emerging trend already supported elsewhere in the corpus, namely that cloud-native attacker behavior often hides inside sanctioned services and normal operational patterns. Downstream researchers should treat [record_id:2273] as a lead for investigating “living off the cloud” taxonomies, cloud-native behavioral detections, and mappings between endpoint LOLBins concepts and cloud control-plane/API abuse.
