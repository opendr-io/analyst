# Topic: Application security

## Executive Summary

The records portray application security as a rapidly expanding field in which familiar web, API, authentication, serialization, and client-side weaknesses remain highly exploitable, while AI agents, LLM-assisted development, and autonomous vulnerability discovery are reshaping both offense and defense. Across the corpus, the strongest recurring message is that “new” platforms often reintroduce old appsec failures: OAuth-style delegation is misimplemented in AI and integration platforms [record_id:3], generative AI developer tools repeat basic security mistakes [record_id:46], agentic systems expose traditional IDOR, sandbox escape, SSRF, and code execution bugs [record_id:2209], and AI-generated code can be syntactically better while still failing basic security tests [record_id:2211].

Classic web security remains central. Records emphasize HTTP desynchronization/request smuggling [record_id:19] [record_id:1989], Unicode normalization bypasses [record_id:13], DOM clobbering at scale [record_id:1947], GraphQL and REST access-control failures [record_id:2018] [record_id:2274], XSS prevention through CSP and Trusted Types [record_id:2532] [record_id:2580], CSRF library flaws [record_id:2300], PHP object injection and deserialization defenses [record_id:14] [record_id:2567], and business-logic vulnerabilities in referral, invitation, paywall, fraud, and subscription flows [record_id:2089] [record_id:2107] [record_id:2254] [record_id:2406].

Authentication, identity, and session security are another major axis. The records cover OAuth/OIDC misconfiguration, token replay, Entra ID authorization mistakes, connection-based OAuth architectures, passkey/WebAuthn/FIDO2 weaknesses, account recovery, risk-based authentication, browser cookie theft, non-human identity governance, and hidden SaaS identities [record_id:3] [record_id:20] [record_id:70] [record_id:126] [record_id:1960] [record_id:1992] [record_id:1997] [record_id:2080] [record_id:2383] [record_id:2418] [record_id:2543] [record_id:2570] [record_id:2572].

AI-related application security is pervasive. The records treat prompt injection as an application architecture problem rather than a single model bug, with talks on indirect prompt injection, “spotlighting,” Targeted Promptware via Google Calendar, malicious MCP servers, guardrail bypass, prompt hardening, AI agent attack matrices, and productionizing secure agentic apps [record_id:58] [record_id:71] [record_id:85] [record_id:144] [record_id:170] [record_id:171] [record_id:1965] [record_id:2207] [record_id:2243] [record_id:2306] [record_id:2313] [record_id:2361] [record_id:2506] [record_id:2554]. In parallel, many records describe LLMs and AI agents as tools for vulnerability discovery, triage, fuzzing, SAST enhancement, exploit validation, synthetic vulnerable app generation, and bug-bounty automation [record_id:24] [record_id:63] [record_id:69] [record_id:72] [record_id:79] [record_id:82] [record_id:122] [record_id:1923] [record_id:2137] [record_id:2182] [record_id:2241] [record_id:2326] [record_id:2329] [record_id:2344] [record_id:2350] [record_id:2374] [record_id:2395].

The evidence base is strongest where records report concrete CVEs, large-scale measurements, open-source tooling, or named vulnerable systems. It is thinner where records are panels, workshops, open discussions, short blurbs, or have empty raw text. Overall, the corpus suggests that modern application security is converging with product security, AI security, identity security, privacy engineering, secure coding education, and vulnerability management.

## Major Themes And Trends

### 1. Old appsec bug classes keep reappearing in new architectures

A recurring theme is that modern platforms often fail at old controls: input parsing, trust boundaries, origin separation, serialization, authorization, and state management. Unicode normalization is presented as a way to bypass WAFs, filters, backend logic, and application assumptions, leading to RCE, XSS, SSTI, open redirects, and response splitting [record_id:13]. HTTP/1.1 request smuggling/desync is framed as still viable on hardened targets, with “wafer-thin mitigations” collapsing under creative analysis [record_id:19] and a DEF CON version making similar claims about mass credential compromise and memory disclosure [record_id:1989]. DOM clobbering research shows client-side code-reuse attacks at scale, with “497 zero-day DOM Clobbering gadgets” in the Tranco Top 5,000 and complete chains in Jupyter, HackMD, and Canvas LMS [record_id:1947].

Serialization remains a core appsec risk. QUACK proposes automatically binding deserialization calls to allowed classes using static duck typing, claiming to block 97% of potentially usable gadget code on average [record_id:14]. A separate PHP object injection talk introduces an Arbitrary Object Instantiation primitive that bypasses `__wakeup()` defenses by triggering dynamic class instantiation outside `unserialize()` [record_id:2567]. AI and ML workflows introduce related risks through model and data file loading: PyTorch’s `torch.load(weights_only=True)` is reported to support TorchScript in a way that led to CVE-2025-32434 RCE [record_id:2], while another record argues that trusted AI formats such as ONNX, HDF5, Feather, YAML, JSON, and NPZ can become payload carriers when deserialized by common ML libraries [record_id:2118].

The records also show that business logic remains difficult to automate and easy to overlook. Referral programs hide race conditions, cookie injection, client-side path traversal, and other logic flaws [record_id:2089]. Invitation systems in Facebook and Snapchat are described as exposing unauthorized access, persistent connections, account blocking, and privacy breaches [record_id:2406]. Recursive Request Exploits weaponize interdependent web requests to bypass authentication, authorization, payment, and entitlement controls in streaming and e-commerce-like flows [record_id:2107]. Fraud systems and live-event fraud prevention appear as application-layer reliability and abuse-prevention problems, with GenAI used for red teaming fraud controls [record_id:1950] and Netflix live events raising fraud mitigation challenges under unpredictable load and behavior [record_id:2254].

### 2. Authorization, OAuth, passkeys, and sessions are still fragile

Identity and access control records repeatedly emphasize the gap between authentication and authorization. The Entra OAuth talk explicitly frames a “common critical misconfiguration” as stemming from difficulty distinguishing authentication from authorization, with access to over 20 internal Microsoft applications claimed as impact [record_id:20]. Connection-based OAuth architectures for AI agents and low-code integration platforms are described as proprietary abstractions that reintroduce classic web vulnerabilities such as session fixation, open redirect, confused deputy, XSS, and cross-window communication attacks [record_id:3]. A BSidesLV OAuth/OIDC session covers missing state parameters, improperly validated discovery documents, token validation failures, and JWT replay with OWASP ZAP [record_id:2383]. Another DEF CON talk claims novel ways to turn Microsoft’s legitimate `login.microsoftonline.com` domain into phishing infrastructure by relying on legitimate functionality [record_id:1960].

Passkeys and FIDO2 are treated as promising but not magic. Android key attestation is analyzed in the context of passwordless banking and bot protection, with PKI vulnerabilities and issues in Google’s open-source X.509 chain parser used to bypass bot protections [record_id:74]. CTRAPS attacks target CTAP and authenticator APIs, including client impersonation and API confusion, with impacts such as credential wiping, unauthorized factory resets, and user tracking [record_id:1992]. “Passkeys Pwned” shows WebAuthn API call proxying to forge registration and authentication responses where sites fail to enforce attestation or metadata checks [record_id:1997]. “Your Passkey is Weak” argues synced passkeys can still be phished and require responsible deployment guidance [record_id:2080].

Session protection is another long-running arms race. Account recovery is framed as an underexamined part of passwordless futures, with a test suite based on recovering accounts at major websites “thousands of times” [record_id:70]. Risk-Based Authentication is evaluated in production 2FA systems against opportunistic, targeted, and advanced attacks [record_id:126], while an older CAMLIS entry concerns IP-based geolocation in 2FA but has no raw explanatory text [record_id:187]. Browser cookie theft remains central: Chrome AppBound encryption, DPAPI, ChaCha20Poly1305, Gecko, WebKit, and upcoming Device Bound Session Cookies are discussed as part of cookie extraction and defense [record_id:2543], and another talk speed-runs the cookie/session defense arms race and Google’s proposed Device Bound Session Credentials [record_id:2570]. Hidden SaaS identities and non-human identities broaden the session/authentication concern beyond human SSO accounts [record_id:2418] [record_id:2572].

### 3. AI application security is being reframed as system and architecture security

The AI records largely reject the idea that prompt injection is just a prompt problem. NCC Group’s agentic AI talk says eliminating prompt injection is “a lost cause” and warns that treating guardrails as first-order controls creates horizontal and vertical privilege escalation vectors [record_id:58]. NVIDIA’s “From Prompts to Pwns” similarly frames agentic systems as dangerous because broader data access, tool use, and autonomous action increase downstream trust risk from malicious input [record_id:85]. Zenity’s “AI Enterprise Compromise” describes AI assistants moving from initial access to persistence, data harvesting, exfiltration, and impact, sometimes via 0-click attacks, and offers a GenAI Attack Matrix [record_id:71].

Indirect prompt injection appears repeatedly. Spotlighting is proposed as a defense that gives models a continuous signal of input provenance and reportedly reduces attack success from over 50% to below 2% in experiments [record_id:144]. Earlier CAMLIS work covers repeated control-character sequences in ChatGPT that circumvent system instructions [record_id:170] and a tutorial on UPIA and XPIA prompt injection taxonomies [record_id:171]. Google Calendar invitation attacks against Gemini agents show how calendar subjects, emails, or Google Docs can hijack agents and invoke tools to delete events, exfiltrate data, control home devices, geolocate users, or launch worms [record_id:1965]. Google Workspace’s own [un]prompted talk describes the “Perfect Storm” of sensitive data access, untrusted content, and external command capability, and references agent hijacking via a calendar invitation plus defense-in-depth with chaining policies, sandboxing, and output sanitization [record_id:2361].

MCP and agent tooling expand the appsec boundary. A malicious MCP server can inject hidden instructions into metadata responses, manipulate Claude Desktop’s reasoning chain, redirect tool behavior to local file reads, or secretly execute code while hiding actions from the user [record_id:2207]. A BSidesLV MCP talk identifies tool poisoning, shared memory exploits, version drift, and line-jumping attacks as inherent risks in the protocol ecosystem [record_id:2554]. Guardrail evasion through state mismatch is demonstrated with ROT13 smuggling where a security system sees only the last message but the chatbot retains chat history [record_id:2243]. Prompt Hardener automates system-prompt evaluation and hardening using techniques such as spotlighting, random sequence enclosure, instruction defense, and role consistency [record_id:2506]. Other records cover prompt patching and “security steerability” [record_id:2199], LLM detection via refusal patterns and flow-breaking [record_id:2188], and general LLM red-team exercises [record_id:2431] [record_id:2478].

A related trend is that AI coding tools create both productivity and security risk. AI-powered developer tools are described as eroding security boundaries, containing excess functionality, and being deployed with elevated permissions [record_id:46]. Veracode’s LLM-generated code study reports that syntax quality improved but security pass rates remained flat, with Java having a 72% failure rate and Python 38% across tested CWEs [record_id:2211]. “Vibe coding” is repeatedly treated as bypassing traditional AppSec controls and producing code that may function but fail security review [record_id:2201] [record_id:2325] [record_id:2373] [record_id:2568]. One mitigation direction is injecting threat models, requirements, and OWASP guidance into AI coding loops via MCP before generation and verifying outputs afterward [record_id:2373]. Another is AI-native security guidance as a service for consistent defensive advice at scale [record_id:2323].

### 4. AI is also transforming vulnerability discovery, validation, and triage

A major research trend is the use of LLMs and agents to augment or automate appsec testing. CodeQL and SAST enhancement records focus on source/sink coverage, data-flow modeling, reflection, native methods, and framework-specific propagation. One Black Hat record claims 191 sources/sinks across 18 frameworks, over 15% more data flows across 5,000 repositories, reproduction of over 50 historical CVEs missed by original CodeQL, and 5 new CVEs [record_id:24]. MTD analyzes microservice applications with gateway-centric entry-point identification, service dependence graphs, and distance-guided taint analysis, reporting 59 high-risk zero-days and 31 CVEs in open-source apps plus 20 industrial vulnerabilities [record_id:69]. “Let LLM Learn” proposes incremental semantic knowledge learning from static-analysis-selected code slices and claims over 70 previously unknown bugs in real-world projects [record_id:72]. Tree-of-AST combines LLMs with AST analysis and a Locate-Trace-Vote methodology to track flows without predefined source patterns [record_id:82].

Autonomous exploitation and validation receive special emphasis. XBOW’s AI agents use robust exploit validators so agents can make many attempts while maintaining “zero false positives,” reportedly finding over 200 zero-days in Docker Hub web apps [record_id:79]. The [un]prompted “Auth-by-One” talk applies strict validators to authentication and authorization flaws, allowing AI agents to exploit access-control bugs in production systems [record_id:2326]. BSidesLV’s Marvin framework trains reinforcement-learning agents to discover logic-based API vulnerabilities in black-box REST and GraphQL environments by modeling state transitions, roles, session tokens, and responses [record_id:2395]. Netflix’s “Source to Sink” pipeline aims to make LLM first-party vulnerability discovery more accurate and actionable by avoiding “crying wolf” false positives [record_id:2374]. FENRIR combines static pre-triage, two-layer LLM validation, and confidence-based human routing; the record claims 100+ AI-infrastructure vulnerabilities and 21 patched CVEs [record_id:2329]. AISLE claims an AI system found 12 OpenSSL zero-days fixed in January 2026 and hundreds of other vulnerabilities in curl, Linux kernel, and wolfSSL [record_id:2350].

Other methods include LLM-assisted binary vulnerability discovery with static analysis and dynamic debugging [record_id:63], iterative retrieval for vulnerable code discovery [record_id:122], BugTraceAI for endpoint discovery, payload generation, SQLMap export, JWT auditing, SSTI, XSS, SQLi, and report generation [record_id:2182], AI-powered vulnerability management using a browser to interpret Semgrep findings and generate tickets [record_id:2241], and AI-assisted fuzzing workflow automation with target identification, harness generation, seed input generation, reproduction, and patching [record_id:2137]. DARPA AIxCC appears as a competition-level attempt to develop cyber reasoning systems that find and fix software vulnerabilities, with several records around finals, winners, and debriefs [record_id:1923] [record_id:2154] [record_id:2156].

### 5. Browser, frontend, and client-side security remain high-leverage

The browser is both a target and a mitigation platform. Browser extension clickjacking shows fake intrusive elements coercing user interaction to leak password-manager data, credit cards, personal data, login credentials, TOTP, and potentially passkeys across 11 password managers [record_id:1851]. Regex-focused hacking includes using regex to break `postMessage` filters and CORS rules, turn redirects into account takeovers, find SSRF opportunities, extract hidden APIs from JavaScript, and locate secrets in GitHub [record_id:1921]. Privacy Sandbox APIs introduce new deanonymization risks via Attribution Reporting debugging reports, destination hijacking, storage limit oracles, and Shared Storage worklet leaks [record_id:2012] [record_id:2030]. Localhost APIs can be attacked from the browser through the “0.0.0.0” flaw, exposing local ports to remote network attacks [record_id:2277].

On defense, multiple records push browser-native protections as a way to eliminate bug classes. Google’s frontend security record describes strict CSP, Trusted Types, and compile-time protections rolled out across hundreds of products accessed by billions of users [record_id:2532]. Javan Rasokat’s workshop and talk argue that CSP v3, Trusted Types, and Sec-Fetch-Metadata can prevent XSS, CSRF, clickjacking, and cross-origin attacks at scale, shifting from endless patching to prevention [record_id:2433] [record_id:2580]. This theme recurs in secure design discussions that emphasize trade-offs, developer needs, and product context [record_id:2280] [record_id:2533].

Mobile and desktop client-side records extend the same theme. Android LaunchAnywhere bypasses show protected activities can still be invoked through BadResolve despite destination checks, with confirmed Google patches and LLM/MCP-assisted codebase search [record_id:1982] [record_id:2138]. Android Zygote injection explains root-enabled code injection into Zygote to hook native and Dalvik code across apps [record_id:2548]. iOS reverse engineering without jailbreaking uses YouTube modding as a case study in app packaging, method swizzling, dynamic analysis, and in-app debugging [record_id:2537]. Trend Micro Mobile Security itself is shown as an attack surface through unauthenticated reports, stored XSS, and memory-level scan manipulation [record_id:2576]. Desktop applications with embedded local LLMs introduce prompt injection, adversarial inputs, plugin ecosystem abuse, unsafe file parsing, and file-format fuzzing risks [record_id:2428].

### 6. API, SaaS, automotive, and data-platform risks are central appsec concerns

Many records focus on APIs and SaaS platforms where centralization magnifies impact. GraphQL is covered both through a Feeld dating app case study in which access-control failures exposed photos, videos, messages, and other personal data [record_id:2018], and through a BSidesSF talk on schema leaks, brute-force risks, and GraphQL “bomb” attacks [record_id:2274]. SaaS attack surface mapping is described as a discipline with novel approaches and an open-source tool release [record_id:2298]. SaaS data platforms are portrayed as high-risk due to the amount of enterprise and user data they process, with four common architectural and logic pitfalls exposing critical data [record_id:2473].

Automotive and transport records show appsec’s real-world control implications. A dealer platform vulnerability allowed creation of a national admin account, customer lookup by VIN, ownership reassignment, remote control functionality, and user impersonation that bypassed 2FA [record_id:2036]. A shorter companion record summarizes API flaws that led to admin access for 1,000+ USA dealers and remote car takeover risk [record_id:2099]. Smart bus research starts from free WiFi, compromises vehicular routers and private networks, manipulates ADAS/APTS communications, steals driver/passenger information, accesses operational data, and penetrates backend API servers [record_id:2121]. IoT router, NAS, AirPlay, and IoT surveillance records similarly show that app-layer endpoints, web consoles, mobile apps, firmware update flows, and backend APIs often define the practical risk [record_id:80] [record_id:1850] [record_id:1934] [record_id:1984] [record_id:1990] [record_id:2111].

### 7. Product security and organizational practice are moving into the appsec core

The records do not treat AppSec as only technical bug hunting. Multiple talks focus on organizational scale, partnerships, policy, education, and product security. “AppSec as Glue” frames security teams as connectors across platform engineering, compliance, threat detection, and other teams [record_id:2252]. “Secure Code Is Critical Infrastructure” positions secure coding and application security as public-good and policy issues [record_id:2061]. CISA’s Secure by Design work and product security policy trends are discussed in terms of memory safety and industry shifts [record_id:2290]. Ochaun Marshall’s “cheat code” distinguishes application security, platform security, and product security and gives steps applicable across maturity levels [record_id:2384]. Shift-left is treated skeptically where noisy tooling creates chaos rather than empowerment [record_id:2287], while “State of AppSec” panels and product security clinics discuss maturity, practice, and serving community organizations [record_id:2293] [record_id:2296]. Secure coding education appears in workshops for high school and software engineering students [record_id:2452].

Bug bounty appears both as research engine and social system. PortSwigger’s desync work references bug bounties as evidence of impact [record_id:19] [record_id:1989]. DEF CON records discuss hacker-triager relationships, severity debates, unclear submissions, gray-area cases, and bounty disputes [record_id:1909], as well as managing bug bounties at scale across Amazon, PayPal, AWS, Shopify, and Splunk [record_id:1928]. Another record highlights serendipitous bug-bounty findings such as blind XSS and accidental receipt of sensitive internal data [record_id:2042]. Android’s bug bounty is discussed from a platform security perspective [record_id:2266], and AI bug bounty programs introduce adversarial prompts, indirect prompt injection, context poisoning, and RAG manipulation as bounty-relevant vulnerability classes [record_id:1930].

## Methods, Tools, And Approaches Discussed

Several records introduce concrete tools or frameworks for application security testing and defense. QUACK automatically fixes deserialization calls by deriving allowed classes through static duck typing and limiting gadget availability [record_id:14]. MTD uses gateway-centric entry analysis, service dependence graphs, and selective context-sensitive taint analysis for microservices [record_id:69]. Hulk dynamically detects DOM clobbering gadgets and generates working exploits end-to-end [record_id:1947]. Marvin provides vulnerable REST and GraphQL environments for evaluating agents that discover logic-based API flaws [record_id:2395]. Marinade injects realistic, exploitable vulnerabilities into production-quality codebases while preserving functionality and attaching validation scripts, enabling scanner benchmarking and red-team training [record_id:2344].

For static and semantic analysis, CodeQL appears repeatedly. One record enhances CodeQL data flow with LLM-discovered source/sink coverage and patches for reflection, native methods, and value passing [record_id:24]. Another maps cryptographic flows from applications and binaries into SBOM-like cryptographic bills of materials with D3 and GPT analysis [record_id:2186]. Netflix’s source-to-sink pipeline focuses on reducing AI-generated false positives in first-party codebases [record_id:2374]. Google’s CodeMender is referenced as part of advancing code security and building safer software [record_id:2321]. Semgrep appears in vulnerability management workflows and secure coding education contexts [record_id:2241] [record_id:2061].

AI-assisted offensive workflows include BinWhisper for LLM-driven binary vulnerability discovery [record_id:63], XBOW-style exploit validators for zero-false-positive web-app finding [record_id:79], Prompt Scan/XBOW claims of 200+ flaws and 1,000+ bug bounty reports [record_id:2084], FENRIR’s multi-stage LLM validation [record_id:2329], AI OpenSSL zero-day discovery [record_id:2350], Promp2Pwn for Pwn2Own bug hunting [record_id:2357], BugTraceAI for bug hunters using OpenRouter, SQLMap, Caido, and Burp Suite workflows [record_id:2182], and offensive modular agents using GPT-4.1, rigging, BBOT, Neo4j, Nuclei, and HTTPX [record_id:2194]. Orion uses LLMs for fuzzing target identification, harness generation, seed generation, reproduction, and patching [record_id:2137].

Prompt and agent security methods include spotlighting [record_id:144], prompt patching and security steerability benchmarks [record_id:2199], Prompt Hardener’s CLI/Web UI for automated prompt evaluation and hardening [record_id:2506], chatbot.ai for testing jailbreaks across multiple models [record_id:2212], llm-trace for diffing AI application workflow prompts and outputs [record_id:2215], MCP-based security-context injection during AI coding [record_id:2373], and architectural controls for agentic trust flows [record_id:58] [record_id:2361]. Prompt||GTFO open discussions also cover context windows, modular code generation, prompt injection vs. jailbreaking, self-hosted models, MCP as REST layers, and local models for security analysis [record_id:2213] [record_id:2234].

Runtime and defensive tools include Logrip for visualizing server logs and identifying automated bot access patterns through hierarchical IP hashing [record_id:37], Finch as a reverse proxy using TLS/HTTP fingerprints such as JA3, JA4, JA4H, and HTTP/2 plus AI-driven dynamic deception [record_id:1948], RASP and tracing for runtime application security [record_id:2310], eBPF tracing to connect product vulnerabilities to security paths [record_id:2264], seccomp profile generation with Kubescape to contain containerized exploits [record_id:2456], and open-source code-to-cloud stacks using Trivy, GitLeaks, Checkov, ZAP, and OpenGrep [record_id:2440]. SSHamble is described as an open-source SSH research and assessment tool [record_id:2100].

Testing and exploitation methods span Burp Suite, OWASP ZAP, SQLMap, CodeQL, fuzzing, reverse engineering, Ghidra disassembly, browser instrumentation, GraphQL scanning, and bug bounty recon. Unicode fuzzing uses Shazzer, recollapse, and Burp ActiveScan++ enhancements [record_id:13]. OAuth token replay uses OWASP ZAP [record_id:2383]. Router exploitation uses Burp Suite requests and Ghidra disassembly [record_id:1990]. Regex is promoted for finding endpoints, hidden APIs, secrets, SSRF opportunities, and bypassing filters [record_id:1921]. Browser security defenses use CSP v3, Trusted Types, and Sec-Fetch-Metadata [record_id:2433] [record_id:2532] [record_id:2580].

## Notable Talks, Records, And Evidence

Several records are especially representative because they combine a clear appsec weakness, named systems, measurable impact, and mitigation or tooling.

* The HTTP desync records are among the clearest examples of mature bug classes remaining dangerous. The Black Hat version claims multiple new desync classes, mass credential compromise, broad target impact, and an open-source toolkit [record_id:19]; the DEF CON version reinforces the same theme with claims of CDN/SaaS impact and memory dumping “heartbleed-style” [record_id:1989].
* DOM clobbering at scale is notable for its quantified web ecosystem impact: 497 zero-day gadgets in Tranco Top 5,000 sites, major client-side libraries affected, over 200 HTML injection sites, exploit chains in popular apps, and 19 CVEs [record_id:1947].
* The microservice taint-analysis record is unusually strong because it describes a concrete method and reports results across 25 open-source and 5 industrial applications, including 59 high-risk zero-days, 31 CVEs, and industrial command/code execution findings [record_id:69].
* The SAST enhancement record gives detailed technical reasons for missed findings—source/sink coverage and language-feature data-flow disruptions—and reports reproducible improvements across 5,000 repositories [record_id:24].
* The OAuth/identity cluster is important because it shows authorization failures across standards, proprietary abstractions, and enterprise deployments: connection-based OAuth for AI and integrations [record_id:3], Entra internal application access [record_id:20], Microsoft login phishing infrastructure [record_id:1960], and OAuth/OIDC token replay [record_id:2383].

## Gaps, Limits, And Open Questions

The largest limitation is that most records are abstracts or short session descriptions. They provide claims but not full technical artifacts, datasets, patches, exploit code, or independent validation. Claims such as “over 100 critical pre-auth vulnerabilities” in Windows HTTP services [record_id:11], “over 200 zero days” found by AI agents [record_id:79], “100+ vulnerabilities” discovered by FENRIR [record_id:2329], and “12 zero-days in OpenSSL” found by AI [record_id:2350] are important but require follow-up with advisories, CVE details, source repositories, papers, or recordings.

Several records are duplicated or overlap. HTTP desync appears in both Black Hat and DEF CON forms [record_id:19] [record_id:1989]. Privacy Sandbox deanonymization appears twice with near-identical text [record_id:2012] [record_id:2030]. Android LaunchAnywhere/BadResolve appears twice [record_id:1982] [record_id:2138]. Automotive dealer API compromise appears in a full and shorter form [record_id:2036] [record_id:2099]. Downstream agents should deduplicate when building timelines or counting findings.

Some records have weak or empty raw text. CAMLIS entries for attack surfaces in computer use agents, PyRIT, LLM package hallucination, red teaming AI red teaming, webshell detection, and IP geolocation 2FA have empty or nearly empty raw fields [record_id:127] [record_id:128] [record_id:132] [record_id:133] [record_id:174] [record_id:187]. The records should not be treated as substantive evidence beyond their presence in the dataset and any external metadata.

Defense evidence is less mature than offense evidence. Many records demonstrate exploitation, bypasses, and tool-assisted discovery; fewer provide rigorous longitudinal evidence that mitigations work at scale. Browser-native defenses [record_id:2532] [record_id:2580], spotlighting [record_id:144], QUACK [record_id:14], seccomp containment [record_id:2456], and architectural controls for AI agents [record_id:58] [record_id:2361] are exceptions, but even these would benefit from independent deployment studies.

Open questions for downstream research include:

* Which AI-assisted vulnerability discovery claims have public CVEs, patches, datasets, or reproducible tooling?
* How should organizations validate AI-generated security findings without drowning in false positives?
* What architectural patterns reliably contain prompt injection when agents can read untrusted data and execute tools?
* Which OAuth/passkey/WebAuthn implementation mistakes are most prevalent in production?
* Can browser-native protections be turned on by default without breaking legacy applications?
* How should AppSec programs integrate product security, AI coding workflows, runtime tracing, and vulnerability management without becoming “security theater” [record_id:2533]?
* What governance is needed for non-human identities, hidden SaaS accounts, and agent credentials as AI automation expands [record_id:2418] [record_id:2572]?
