# Topic: Machine learning model security

## Executive Summary

The records present machine learning model security as a broad, rapidly expanding field that has moved beyond classic adversarial examples into a full lifecycle problem: model file formats and deserialization, supply-chain trust, LLM backdoors and jailbreaks, data poisoning, model extraction and inversion, evaluation methodology, operational monitoring, and secure deployment of AI agents. The strongest evidence clusters around three recurring concerns.

First, machine learning infrastructure itself is increasingly treated as an attack surface. Multiple 2025 records focus on model-loading and serialization risks, including PyTorch/TorchScript remote code execution despite `weights_only=True` [record_id:2] [record_id:1968], static model scanner bypasses for unsafe formats such as Pickle and Keras [record_id:86], and code execution through trusted AI workflow formats such as ONNX, HDF5, Feather, YAML, JSON, and NPZ [record_id:2118]. These records frame ML supply-chain security as a practical deployment risk rather than a theoretical concern.

Second, adversarial machine learning remains a dominant throughline, especially in cybersecurity contexts. Records cover malware detector evasion, phishing webpage detector attacks, perceptual hashing bypasses, algorithmic trading policy manipulation, autonomous vehicle teleoperation fuzzing, CAN-bus anomaly detection, graphical content moderation evasion, and financial-report manipulation [record_id:165] [record_id:203] [record_id:212] [record_id:55] [record_id:2102] [record_id:2449] [record_id:124]. Older CAMLIS records show sustained interest in adversarial malware classification, including attacks on PE classifiers, poisoning, backdoors, uncertainty, interpretability, and robust evaluation [record_id:213] [record_id:220] [record_id:227] [record_id:230] [record_id:249].

Third, the LLM era has shifted model security toward prompt-level and model-internal failure modes: universal prompt triggers, multi-turn jailbreak frameworks, refusal-direction defenses, graph-level LLM backdoors, white-box safety analysis, typographic prompt injection in VLMs, textual obfuscation attacks, and production prompt-injection detectors for browser agents [record_id:18] [record_id:112] [record_id:113] [record_id:114] [record_id:2382] [record_id:115] [record_id:2375] [record_id:2363]. These records show a trend from output-only testing toward internal-state analysis, mechanistic interpretability, and model-specific monitoring [record_id:148] [record_id:150] [record_id:2346].

The evidence base is strongest where abstracts provide concrete methods, datasets, metrics, or claimed results, especially malware ML, model-loading vulnerabilities, and LLM attack/defense work. It is thinner for records with empty raw text or short talk listings, such as some keynote, panel, and tool-session entries [record_id:128] [record_id:131] [record_id:147] [record_id:161] [record_id:178] [record_id:180] [record_id:190] [record_id:2371].

## Research Landscape

The corpus spans Black Hat, DEF CON, CAMLIS, BSides, Prompt||GTFO, and [un]prompted events from 2017 through 2026. CAMLIS dominates the historical portion and provides the deepest sustained body of work on ML for cybersecurity, especially malware classification, security evaluation, adversarial robustness, feature engineering, uncertainty, and datasets [record_id:198] [record_id:210] [record_id:220] [record_id:238] [record_id:241] [record_id:248]. Black Hat and DEF CON 2025 records contribute more recent infrastructure and offensive tradecraft themes, especially AI model file formats, PyTorch deserialization, Apple on-device AI framework abuse, and model-scanner limitations [record_id:2] [record_id:4] [record_id:86] [record_id:1968] [record_id:2118]. Prompt||GTFO and [un]prompted records emphasize LLM prompt attacks, agent security, mechanistic interpretability, prompt patching, and practical AI red-teaming workflows [record_id:2179] [record_id:2199] [record_id:2212] [record_id:2346] [record_id:2363].

The records collectively treat “machine learning model security” in two overlapping ways. One meaning is the security of ML systems themselves: model theft, extraction, inversion, poisoning, backdoors, deserialization, and unsafe model loading [record_id:167] [record_id:1943] [record_id:227] [record_id:2503] [record_id:2118]. The other is the security reliability of ML models used for cyber defense: malware classifiers, URL detectors, SOC alert triage, vulnerability detection, authentication anomaly detection, CAN-bus anomaly detection, and AI-assisted security workflows [record_id:141] [record_id:168] [record_id:175] [record_id:184] [record_id:223] [record_id:2376].

A notable temporal shift is visible. Earlier records from 2017–2019 focus heavily on making ML work for security tasks and understanding its brittleness: hyperparameter optimization, feature engineering, malware visualization, EMBER dataset improvements, tree-model vulnerability analysis, and adversarial malware examples [record_id:256] [record_id:224] [record_id:218] [record_id:226] [record_id:249] [record_id:213]. Records from 2021–2023 broaden into model extraction, open-set recognition, side-channel robustness, benchmarking, and LLM capabilities [record_id:202] [record_id:167] [record_id:164] [record_id:168]. Records from 2024–2026 increasingly focus on foundation models, LLM safety mechanisms, AI agents, model supply chains, and production deployment risks [record_id:135] [record_id:148] [record_id:150] [record_id:151] [record_id:2363] [record_id:2382].

## Major Themes And Trends

### 1. Model-loading, serialization, and AI supply-chain risks are becoming first-class security issues

The most direct infrastructure-security records argue that ML model artifacts are executable or execution-adjacent assets. The PyTorch/TorchScript research states that `torch.load` with `weights_only=True`, presented in documentation as safer than full pickle loading, still supports TorchScript and led researchers to discover vulnerabilities culminating in RCE and CVE-2025-32434 [record_id:2] [record_id:1968]. This is important because it challenges a widely accepted mitigation boundary: “weights-only” loading is not necessarily a complete deserialization defense.

A related Black Hat record argues that many AI frameworks embed unsafe serialization formats such as Pickle and lambda functions into model formats, and that static model scanners cannot fully emulate the loading process [record_id:86]. The record claims the researchers created and identified “dozens of examples” undetected by existing scanners, including malicious models found in the wild, and proposes dynamic analysis as a more comprehensive mitigation approach [record_id:86]. DEF CON work on AI file formats expands the concern beyond explicitly dangerous Pickle files to trusted formats such as `.onnx`, `.h5`, `.npz`, Feather, YAML, JSON, and HDF5, claiming code execution can follow from default library behavior in common ML workflows [record_id:2118].

Together, these records suggest a trend: model security is converging with software supply-chain security. The dangerous object is no longer only source code or a package dependency; it is the model artifact, the dataset container, the weight file, the vector store, and the auto-loading pipeline.

### 2. Adversarial ML remains highly practical in cyber defense settings

The records repeatedly show ML classifiers in security domains being attacked through domain-preserving manipulations. In phishing detection, one CAMLIS paper proposes functionality- and rendering-preserving HTML manipulations selected by query-efficient black-box optimization, claiming state-of-the-art ML phishing webpage detectors can be “razed to the ground” with about 20 queries [record_id:165]. In malware detection, multiple records focus on PE-file modifications, adversarial examples, backdoor poisoning, and red-team generated adversarial ransomware behavior [record_id:213] [record_id:220] [record_id:227] [record_id:230]. BSidesLV 2025 adds a practitioner-oriented example in which minor modifications to well-known malware samples reduced detectability by AI-based and traditional detection methods without changing functionality [record_id:2420].

This adversarial theme extends beyond malware. Perceptual hashing for client-side scanning is evaluated under black-box and white-box detection-avoidance attacks, with the record claiming more than 99.9% of images were successfully attacked while preserving content [record_id:212]. Deep reinforcement learning trading policies are framed as susceptible to active test-time manipulation and passive adversarial policy imitation [record_id:203]. Autonomous vehicle teleoperation is tested with an “Adversarial Scenario Fuzzer” that generates malicious trajectory suggestions, conflicting guidance signals, and environmental perturbations to evaluate resilience [record_id:55]. A financial-reporting AML record describes manipulating financial statements to inflate earnings while reducing fraud scores [record_id:124]. A CAN-bus anomaly detection talk presents deep sequence learning over raw CAN payloads to detect stealthy contextual attacks without signal decoding [record_id:2102].

The trend is that adversarial ML is treated less as image-classifier novelty and more as operational robustness testing for high-stakes domains: endpoint security, fraud, E2EE scanning, vehicles, trading, and content moderation.

### 3. LLM security is moving from prompt attacks toward internal mechanisms and backdoors

Several records focus on prompt injection and jailbreaks, but the most interesting trend is the movement from surface prompts to model internals. A Black Hat 2025 record describes a gradient-based prompt-injection technique that discovers “universal and context-independent triggers” forcing open-source LLMs to produce attacker-chosen text regardless of original prompt or task, with demonstrations against LLM applications leading to RCE [record_id:18]. CAMLIS 2025 includes HarmNet, a modular framework for adaptive multi-turn jailbreak queries with higher claimed attack success than prior methods [record_id:112], and LLM salting, a defense that rotates the internal refusal direction to invalidate jailbreak prompts such as GCG without degrading utility [record_id:113].

Backdoor work is especially prominent. ShadowLogic modifies white-box LLM computational graphs to inject hidden backdoors activated by a secret trigger phrase, producing uncensored responses and exposing graph-level vulnerabilities [record_id:114]. A 2024 CAMLIS record observes that LLM backdoor activations “stick together” and proposes targeted noising of neurons to neutralize Trojans while preserving LAMBADA accuracy on Pythia and Llama2 [record_id:150]. A 2026 [un]prompted record argues that output-only testing can miss backdoors that weaken refusal behavior internally while outputs still appear compliant, and proposes white-box analysis of internal refusal signals [record_id:2382]. A related 2024 defense uses residual stream activation analysis between transformer layers to classify attack prompts and evaluate safety fine-tuning [record_id:148]. A [un]prompted talk generalizes this into “Glass-Box Security,” using mechanistic interpretability and latent-space geometry as an “Internal EDR” for AI agents [record_id:2346].

The implication is that LLM security evaluation is no longer only a benchmark of refusals or jailbreak success. Several records argue for monitoring residual streams, neuron activations, refusal directions, latent geometry, and computational graphs [record_id:148] [record_id:150] [record_id:2346] [record_id:2382].

### 4. Evaluation, benchmarking, and dataset quality are recurring foundations

A substantial portion of the corpus addresses how to evaluate cybersecurity ML models under noisy labels, class imbalance, changing distributions, or costly errors. Firenze uses weak signals from domain-expert “markers” and statistical hypothesis testing to compare ML models where labels are noisy, limited, or biased [record_id:184]. A cost-aware alternative to F1, Cscore, explicitly models unequal false-positive and false-negative costs and reports average cost savings of 49% across five cybersecurity datasets [record_id:142]. Earlier work on website content classification similarly addresses asymmetric misclassification costs through cost-cluster auxiliary losses [record_id:245], while uncertainty estimation for binary classifiers argues that raw output score is often a poor uncertainty proxy and proposes Bayesian approximation using deep networks to estimate hierarchical Bayesian parameters [record_id:248].

Dataset quality is a major concern. An analysis of C/C++ datasets for ML-assisted vulnerability detection finds divergence from a new large “Wild C” dataset, missing vocabulary, duplication, and a Juliet split issue where random 80/20 splitting yields roughly 58% overlap of test with training due to augmentations [record_id:198]. Malware benchmark work includes MalDICT datasets for malware behaviors, platforms, exploited vulnerabilities, and packers using ClarAVy across 90 AV products and about 5.5 million malicious files [record_id:160], EMBER updates that aim to create a harder 2018 test set while reducing outliers and duplications [record_id:226], and train/test split generation methods designed to make small effect sizes detectable in malware research [record_id:158]. VirusTotal label evolution and AV consensus are studied both for label generation over time and for modeling correlations among AV engines [record_id:240] [record_id:200].

The LLM era adds new benchmarks and model comparisons. Cybersecurity-specific LLM benchmarks evaluate SQL translation, incident severity prediction, and IOC summarization across GPT-4, MPT-30B-Instruct, Claude, and others [record_id:168]. BrowseSafe-Bench embeds prompt-injection attacks in high-entropy HTML pages for production browser agents, with a fine-tuned Qwen-30B mixture-of-experts model reportedly achieving about 0.91 F1 under sub-100ms latency [record_id:2363]. Veracode research tests 100 LLMs across Java, .NET, Python, and JavaScript against four CWEs and finds syntax quality improved but secure-code pass rates remained flat [record_id:2211]. A short [un]prompted entry asks whether traditional ML or LLMs classify better, but raw evidence is limited to the title and speaker description [record_id:2371].

### 5. Model extraction, inversion, privacy, and data control remain persistent concerns

The corpus includes multiple records where privacy failure comes from model behavior rather than database exposure. A 2018 record discusses black-box model inversion and inferring model families from deployed black-box APIs using decision topologies [record_id:234]. A 2023 record introduces Model Leeching, an extraction attack that distills task-specific knowledge from ChatGPT-3.5-Turbo into a smaller model, claiming 73% exact-match similarity and SQuAD EM/F1 of 75%/87% for about $50 in API cost [record_id:167]. A DEF CON 33 talk extends privacy risk to AI systems using LLMs and RAG workflows, including model inversion against fine-tuned models and embedding inversion against vector databases to extract PII and social security numbers [record_id:1943].

Defensive or architectural privacy records include Curl, an MPC framework for private LLM evaluation using wavelet-encoded lookup tables, reporting reduced rounds, communication, latency, and a proof for probabilistic truncation protocols in the stand-alone model [record_id:149]. AdapterSwap proposes dynamically composed low-rank adapters to support continual learning, user-based access controls, and document deletion guarantees while avoiding forgetting [record_id:151]. XAI itself is also framed as a privacy attack surface: counterfactual explanations may enable membership inference, model extraction, evasion, poisoning, and backdoor attacks [record_id:204].

### 6. ML for cyber operations is both an asset and an attack surface

Several records are not primarily about attacking models but about deploying ML in security operations. These include graph neural networks for malicious Azure AD sign-in detection [record_id:175], machine learning for malicious URL detection using lexical features and random forests [record_id:223], SOC alert classification explanations for analysts [record_id:244], multi-stage classification for distributed security monitoring [record_id:251], malware disassembly annotation through neural machine translation [record_id:207], and vulnerability detection lessons from 125K Linux kernel commits [record_id:2376]. These records highlight practical deployment issues: human-in-the-loop explanation, out-of-distribution samples, latency, data access, analyst trust, and cost tradeoffs.

At the same time, offensive-use records show ML capabilities being weaponized. One Black Hat talk describes MLArc, a CoreML-based C2 framework that embeds and executes encrypted shellcode inside AI models and uses Vision and AVFoundation for payload/key concealment through images and audio [record_id:4]. Another describes reinforcement-learning fine-tuning of a 7B-parameter model for evasive malware development, explicitly targeting AV/EDR evasion tasks [record_id:62]. BSidesLV work describes fine-tuning a generative AI model to become a stealthy C2 and exfiltration channel after a victim trains on attacker-crafted data [record_id:2573]. A BSidesSF listing describes fine-tuning Llama 3.1 into an unguarded “destructive hacking robot,” but the raw text is brief and rhetorical rather than methodologically detailed [record_id:2247].

## Methods, Tools, And Approaches Discussed

The records discuss a wide range of methods. For adversarial testing, examples include query-efficient black-box optimization over HTML manipulations for phishing detector evasion [record_id:165], reinforcement learning over PE-file perturbations using gym-malware to attempt Meterpreter evasion [record_id:213], adversarial scenario fuzzing for autonomous teleoperation commands [record_id:55], and black-box/white-box perturbation attacks against perceptual hashing algorithms [record_id:212]. TreeHuggr analyzes split points in tree ensembles to identify vulnerable regions of feature space rather than merely producing a single evasion example [record_id:249]. Bayesian cover trees monitor adversarial drift and claim black-box attacks induce measurable drift even when attackers try to hide in benign traffic [record_id:211].

For malware model robustness and interpretability, records include hidden-representation regularization that encourages original PE files and ASCII-overlay-modified “twins” to share internal representations [record_id:220], SHAP-guided backdoor poisoning against malware classifiers [record_id:227], integrated gradients for ransomware behavior attribution and adversarial analysis [record_id:230], Grad-CAM overlays on image-transformed malware disassembly [record_id:218], activation analysis of byte-based malware networks with SHAP and reverse-engineer validation [record_id:238], and structure/semantics-aware Vision Transformer malware classifiers that incorporate PE section information and evaluate robustness to evasion [record_id:141].

For model evaluation, methods include weak-signal markers and regions of interest in Firenze [record_id:184], cost-aware Cscore based on precision/recall and a false-positive/false-negative cost ratio [record_id:142], cost-cluster auxiliary losses for multi-class misclassification asymmetry [record_id:245], Bayesian uncertainty estimation for binary classifiers [record_id:248], configurable malware train/test splits based on malware family information and secondary-model generalization rates [record_id:158], and temporal modeling of VirusTotal labels and AV consensus [record_id:240] [record_id:200].

For LLM and agent security, tools and approaches include PyRIT, described as a model- and platform-agnostic open-source framework for red-teaming generative AI systems across modalities [record_id:135]. A 2025 CAMLIS entry specifically names accelerating AI red-team operations with PyRIT, but its raw text is empty, so the substantive evidence comes from the 2024 PyRIT record [record_id:128] [record_id:135]. HarmNet constructs adaptive multi-turn jailbreaks [record_id:112]. LLM salting rotates refusal directions [record_id:113]. Text2VLM converts text-only datasets into multimodal formats for evaluating typographic prompt injection in visual language models [record_id:115]. BrowseSafe uses a fine-tuned MoE model and production feedback to detect prompt injection in browser agents [record_id:2363]. Prompt patching introduces “security steerability” as a benchmark for adherence to system-prompt restrictions under conflicting user instructions [record_id:2199]. A multi-model jailbreak testing tool, “chatbot.ai,” runs one prompt across many model/configuration combinations to test transferability [record_id:2212].

For model internals, methods include residual stream activation analysis [record_id:148], targeted neuron noising for Trojan mitigation [record_id:150], graph-level computational modifications for ShadowLogic backdoors [record_id:114], white-box refusal-signal analysis for backdoor detection [record_id:2382], and mechanistic-interpretability “semantic tripwires” for internal AI-agent monitoring [record_id:2346].

For privacy-preserving and data-governed ML, Curl uses MPC with wavelet-compressed lookup tables for nonlinear functions in LLMs [record_id:149], while AdapterSwap organizes knowledge into dynamically composed low-rank adapters for continual learning, access control, and deletion [record_id:151]. For visualization and education, GraphLeak visualizes how poisoned training data alters network structure and lets users compare clean versus poisoned data interactively [record_id:2503].

## Notable Talks, Records, And Evidence

The PyTorch/TorchScript records are among the most important because they make a concrete vulnerability claim: `torch.load(weights_only=True)` supporting TorchScript led to vulnerabilities and RCE, acknowledged by PyTorch as CVE-2025-32434 [record_id:2] [record_id:1968]. These are representative of the shift from “ML model behavior” to “ML framework exploitation.”

The model-scanner bypass talk is also central because it challenges a common defensive control. It argues that static analysis cannot accurately emulate model-loading behavior and that malicious models can evade existing scanners, motivating dynamic approaches across the model lifecycle [record_id:86]. The AI file-format code-execution talk broadens this to common workflow containers and libraries, making it relevant to ML engineering, not just model hosting [record_id:2118].

For adversarial ML in security, the phishing webpage detector paper is a strong representative because it clearly states the attacker constraints: preserve maliciousness and visual rendering, use fine-grained HTML manipulations, and optimize under black-box query limits [record_id:165]. The perceptual hashing record is similarly strong because it includes large-scale evaluation claims and privacy implications of raising detection thresholds [record_id:212]. The backdoor poisoning malware record is notable for adapting image-classifier backdoor threat models to malware, using SHAP values to select watermark features, and evaluating mitigation through anomaly detection and dataset cleansing [record_id:227].

For LLM model security, the universal trigger record is notable because it describes discrete gradient descent over diverse instruction datasets to discover prompt triggers that generalize across context and can lead to RCE in LLM applications [record_id:18]. ShadowLogic is notable because it moves backdoors into computational-graph modifications rather than prompt or dataset triggers [record_id:114]. The white-box backdoor detection talk is important because it explicitly argues that correct outputs do not imply intact safety mechanisms [record_id:2382]. BrowseSafe is practically important because it claims production deployment and discusses latency, false positives, real-world attack patterns, and adversarial evolution in browser agents [record_id:2363].

For evaluation and datasets, Firenze is representative of security ML deployment realities: labels are noisy and weak signals from experts can be more useful than conventional metrics alone [record_id:184]. Cscore is representative of the recurring cost-sensitivity theme [record_id:142]. The C/C++ dataset analysis is notable because it identifies data leakage and representativeness failures in vulnerability-detection datasets [record_id:198]. MalDICT and EMBER improvements are important because they provide or improve public-ish benchmarks for malware ML [record_id:160] [record_id:226].

For malware ML explainability and robust classification, the activation-analysis work on byte-based malware classifiers is notable for bridging low-level neural activations to reverse-engineer semantics on a dataset of more than 15 million executables [record_id:238]. Structure and semantics-aware malware classification with Vision Transformers is notable because it claims lightweight structural information such as PE sections lets deep models match traditional malware classifiers at commercial scale [record_id:141]. Kampff is notable as an accessible adversarial malware defense using a portfolio of classifiers attached to a primary malware classifier [record_id:210].

For privacy and extraction, Model Leeching is notable for concrete extraction-cost and performance claims against ChatGPT-3.5-Turbo [record_id:167]. The DEF CON “shadow data” record is representative of the newer RAG/vector-database privacy issue, combining model inversion and embedding inversion [record_id:1943]. Curl and AdapterSwap represent more defensive architectural responses to privacy, confidentiality, access control, and deletion [record_id:149] [record_id:151].

## Gaps, Limits, And Open Questions

The corpus is rich in talk abstracts and claimed results but thin in independently verifiable experimental details. Many records are conference previews rather than full papers, so downstream researchers should treat numerical claims as leads rather than settled findings unless they can locate papers, code, datasets, or recordings. This is especially true for high-impact claims such as RCE via ML file loading, model-scanner bypasses, universal LLM triggers, and production prompt-injection detection [record_id:2] [record_id:86] [record_id:18] [record_id:2363].

Several entries have empty or minimal raw text. Records for PyRIT 2025, ScamAgents, a 2024 keynote panel, a 2023 generative AI keynote, ARNIE, an algorithmic bias bounty, a multi-tenant ML workflow platform, and traditional ML versus LLM classification provide little or no substantive evidence beyond metadata/title-level context [record_id:128] [record_id:131] [record_id:147] [record_id:161] [record_id:178] [record_id:180] [record_id:190] [record_id:2371]. These should not be used to support detailed claims without outside corroboration.

A persistent open question is how to validate defenses under adaptive attackers. Records propose static-to-dynamic model scanning, targeted neuron noising, LLM salting, prompt patching, BrowseSafe, Bayesian drift monitoring, and adversarial retraining or regularization [record_id:86] [record_id:150] [record_id:113] [record_id:2199] [record_id:2363] [record_id:211] [record_id:220]. But the corpus does not provide a unified evaluation standard for comparing these defenses under repeated adaptation.

Another gap is operational integration. Many records identify vulnerabilities or propose tools, but fewer discuss long-term deployment: who owns model artifact review, how model scanners are integrated into CI/CD, how often internal LLM safety signals are monitored, how false positives are handled, and how feedback loops avoid poisoning or privacy leakage. BrowseSafe gives one of the more concrete production accounts [record_id:2363], and SOC-oriented explainability records address analyst workflow [record_id:244], but this remains underdeveloped.

There is also tension between model capability and model security. Records on specialist malware-development models, fine-tuned C2, Apple AI C2, destructive hacking agents, and LLM-generated insecure code all suggest that model capabilities can amplify offensive operations or insecure development practices [record_id:62] [record_id:2573] [record_id:4] [record_id:2247] [record_id:2211]. The corpus does not resolve how to distinguish legitimate security research, red-team tooling, and harmful automation at governance or technical-control levels.

Finally, the dataset and benchmark records show that the field continues to struggle with representativeness, duplication, temporal drift, and label quality [record_id:198] [record_id:240] [record_id:158] [record_id:160]. This raises an open question for nearly every attack and defense claim: does it generalize beyond the benchmark, the family, the model, or the lab setting?

## Coverage Addendum

The missing Prompt||GTFO 2026 record adds a distinct robustness angle not fully represented in the existing report: **mathematical equivalence sensitivity in LLMs**. Michael Shalyt’s “Mathematical Vulnerabilities in LLMs” demonstrates that several LLMs can respond inconsistently or incorrectly when given semantically identical mathematical expressions formatted in different ways. The examples include adding meaningless zeros to numbers, switching multiplication notation between asterisks and LaTeX `\cdots`, and replacing trigonometric expressions with equivalent forms such as `1/sin` versus cosecant. The reported effects include wrong answers, crashes, unexpected behavior, and spontaneous language switching across models including GPT-4o, Gemini Flash, Grok, and DeepSeek R1 [record_id:2230].

This record broadens the “model behavior and robustness” theme beyond adversarial malware, prompt jailbreaks, and model-loading vulnerabilities.
