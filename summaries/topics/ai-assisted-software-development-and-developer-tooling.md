# Topic: AI-assisted software development and developer tooling

## Executive Summary

The records collectively portray AI-assisted software development as a rapidly maturing but unsettled field. The corpus is dominated by conference talks from Prompt||GTFO 2026, [un]prompted 2026, and BSidesLV 2025, with repeated attention to “vibe coding,” coding agents, AI IDEs, requirements engineering, agentic security tooling, and LLM-assisted code analysis. Across the records, AI coding tools are presented as capable of accelerating real development tasks—from building games and browser extensions to optimizing PyTorch code and generating conference tools—but also as introducing new reliability, security, governance, and evaluation problems.

A major trend is the normalization of AI as part of the development loop. Several records describe practitioners using Claude Code, Cursor, Gemini, GPT-5, and related tools to build or modify software interactively [record_id:2183] [record_id:2189] [record_id:2223] [record_id:2227] [record_id:2352]. At the same time, the security-focused records argue that these tools expand the attack surface of software development environments, bypass traditional application security controls, and require new forms of policy enforcement, evaluation, and context injection [record_id:2335] [record_id:2345] [record_id:2373] [record_id:2568].

The strongest evidence clusters around live-demo-oriented conference talks. These records tend to describe concrete tools, workflows, or demonstrations: using AI agents to optimize a PyTorch bottleneck [record_id:2223], building a Chrome extension through iterative prompting [record_id:2189], designing a Gemini-based requirements engineering assistant [record_id:2227], enforcing deterministic policies around coding-agent actions with Cedar [record_id:2345], injecting threat-model and OWASP context into AI coding loops through an MCP server [record_id:2373], and improving LLM-driven vulnerability discovery through an agentic source-to-sink pipeline [record_id:2374]. The evidence is weaker where descriptions are short, abstract, or programmatic, such as the micro-talk collection on tools created at the conference [record_id:2352].

Overall, the records suggest a research landscape moving beyond “can AI write code?” toward more operational questions: How should teams structure projects so AI-generated code remains maintainable? How should organizations evaluate agent effectiveness? How can security requirements, threat models, and deterministic controls be integrated into agentic development workflows? And what risks arise when development becomes faster, cheaper, and more abstracted from traditional engineering practice?

## Research Landscape

The records are primarily conference talk summaries, not papers or long-form technical reports. Most come from 2026 security- and AI-focused events—Prompt||GTFO and [un]prompted 2026—with one BSidesLV 2025 talk providing an earlier security-oriented critique of vibe coding [record_id:2568]. Because the sources are talk abstracts and short descriptions, they provide useful signals about practitioner concerns, emerging toolchains, and demonstrated workflows, but generally do not include detailed experimental methods, quantitative evaluations, or reproducible artifacts.

The Prompt||GTFO records emphasize hands-on AI-assisted development and practical experimentation. Imri Goldberg’s “Vibe Coding a Game with the Kids and Godot” describes building a space shooter with an 8-year-old using Godot and AI coding agents including Claude Code and Cursor [record_id:2183]. Gadi Evron’s “Mail Goggles” talk similarly focuses on iterative AI-assisted implementation of a Chrome extension using GPT-5, including specialized developer and tester prompts to handle bugs, UI issues, and Manifest 3 complications [record_id:2189]. Halvar Flake’s PyTorch optimization talk is especially concrete: it describes using Claude Code and Gemini CLI to profile a slow neural-network visualization script and change the implementation so hashing activation patterns happens during the forward pass on the GPU, improving performance from about one minute to about half a second per frame [record_id:2223]. Allan Stojanovic’s requirements engineering talk broadens the topic from code generation to upstream software specification, using a custom Gemini “gem” to ask structured questions, track conflicts, and generate formal requirements documents [record_id:2227].

The [un]prompted 2026 records lean more heavily toward security, governance, and evaluation of agentic development. Several talks treat coding agents not merely as productivity tools but as powerful actors with access to codebases, terminals, files, and deployment-relevant workflows. Piotr Ryciak’s talk frames AI IDEs and coding agents as an expanded attack surface, citing systems such as OpenAI Codex, Amazon Kiro, Google Antigravity, Cursor, and others, and organizing exploitation patterns by trigger model such as zero-click, one-click, autorun, and time-delayed execution [record_id:2335]. Matt Maisel’s Cedar talk proposes a deterministic reference monitor to intercept shell commands, file reads, and other coding-agent actions, with compatibility claims for Cursor, Claude Code, and GitHub Copilot CLI [record_id:2345]. Srajan Gupta’s talk proposes an MCP server that injects security context before code generation and verifies generated output afterward [record_id:2373]. Scott Behrens and Justice Cassel describe improving LLM first-party vulnerability discovery with an agentic source-to-sink pipeline designed to reduce noisy false positives [record_id:2374].

Several records connect AI-assisted development to security programs and agent evaluation. Google’s “Evaluating Threats & Automating Defense” talk includes CodeMender as an example of tools helping build “intrinsically safer software” in the context of AI security strategy and code security [record_id:2321]. Joshua Saxe’s Meta talk argues that as AI coding tools lower the cost of building security agents, the hard part becomes measuring whether agents work against real attacks and unseen vulnerabilities; it describes moving from naive precision/recall to multi-dimensional evaluation of reasoning quality, evidence gathering, and tool-calling logic, and using genetic algorithms plus AI coding tools for automated agent improvement [record_id:2322]. OpenAI’s “Code Is Free” talk explicitly states that engineers are using LLMs to write code and asks why they are not also using them for security, advocating “zero-friction” security improvements and model-assisted security invariants [record_id:2325].

This corpus therefore covers both constructive and critical perspectives. It includes enthusiastic examples of AI accelerating development, security-oriented approaches for making AI-generated or AI-assisted software safer, and cautionary accounts of vulnerabilities and organizational risks introduced by vibe coding.

## Major Themes And Trends

### 1. “Vibe coding” as a practical development mode

A central recurring theme is “vibe coding”: building software through conversational prompting, iterative refinement, and AI-generated code rather than conventional hand-authored software engineering. The records do not present a single settled definition, but they collectively show vibe coding being used for games, browser extensions, conference tooling, and general software development workflows.

The most concrete positive example is the Godot game project, where Imri Goldberg describes vibe coding a space shooter with his 8-year-old son using Claude Code and Cursor [record_id:2183]. The record is notable because it frames AI coding not as a toy demo but as a path toward shipping a game on Steam as a “purely vibe-coded project” [record_id:2183]. It also highlights lessons about architecture, testing, and modular design, suggesting that even when code is generated by agents, software engineering structure remains important for stability [record_id:2183].

The “Mail Goggles” Chrome extension similarly shows vibe-style iterative development in a browser-extension context. Gadi Evron used GPT-5 to build a math-gated email-sending extension, creating specialized developer and tester prompts to address bugs, UI problems, and Chrome Manifest 3 issues [record_id:2189]. The partial live-demo failure is important evidence: it suggests that AI-assisted coding can produce working systems, but reliability and demo robustness remain uneven [record_id:2189].

The [un]prompted “Vibecoded” record indicates that participants built tools at the conference to improve the conference itself, although the abstract is sparse and gives few technical details [record_id:2352]. Its value is mainly as evidence that vibe coding had become a community practice and conference activity, not only an individual development approach.

The BSidesLV “dark side” talk provides a counterweight. It defines vibe coding as generating software using AI instead of traditional software engineering practices and warns that this introduces data leakage, model manipulation, social engineering, insecure generated code, and organizational risks [record_id:2568]. That record is more skeptical than the Prompt||GTFO demos: it argues that simple-prompt-generated code can appear functional while repeatedly failing basic security scrutiny, and it warns about developers becoming disconnected from underlying computing principles [record_id:2568].

### 2. AI coding tools are shifting from code generation to end-to-end workflows

The corpus shows AI assistance extending beyond isolated code snippets. The tools and workflows described span requirements engineering, implementation, testing, profiling, performance optimization, vulnerability discovery, policy enforcement, and security review.

Requirements engineering appears explicitly in Allan Stojanovic’s Gemini-based workflow. The custom Gemini prompt is designed to reduce “AI chattiness,” ask questions one at a time, track conflicts between requirements, and produce formal requirements documents with version control for export to Google Docs and potential use in vibe coding workflows [record_id:2227]. This points to an upstream integration of AI into software development: before code is generated, the AI helps clarify, structure, and version requirements.

Performance optimization appears in Halvar Flake’s PyTorch example. Rather than simply asking the model to rewrite code, the workflow includes profiling, identifying GPU-to-CPU transfer overhead, and modifying the architecture using PyTorch forward hooks so hashes are computed during the forward pass on the GPU [record_id:2223]. The reported performance improvement—from roughly one minute to roughly 0.5 seconds per frame—is one of the clearest quantitative claims in the corpus [record_id:2223].

Security review and vulnerability discovery appear in several records. Netflix’s “Source to Sink” talk describes an agentic pipeline for first-party vulnerability discovery that is intended to reduce LLM over-reporting, summarized by the phrase “thinks before it screams” [record_id:2374]. The problem statement is practical: teams have had to mass-close large numbers of AI-generated findings, and the talk focuses on improving accuracy and actionability in real-world codebases [record_id:2374].

The OpenAI “Code Is Free” talk frames LLMs as tools not only for generating application code but also for creating security invariants and improving software security with low-friction engineering changes [record_id:2325]. Google’s CodeMender-related talk similarly presents AI as part of automated defense and code security, with a goal of “intrinsically safer software” [record_id:2321].

### 3. Security controls are being moved into the AI coding loop

Several records argue that traditional AppSec workflows are not sufficient when developers use AI IDEs and coding agents. These talks propose injecting security context, monitoring agent actions, and designing security-specific agent workflows.

Srajan Gupta’s talk is the clearest example of pre- and post-generation security integration. It argues that vibe coding with tools like Cursor is fast but can quietly bypass traditional AppSec controls [record_id:2373]. The proposed method is an MCP server that, before code is generated, pulls threat models, security requirements, and OWASP guidance for the task; after generation, it verifies the output for vulnerabilities and standards compliance [record_id:2373]. This turns AppSec from a downstream review stage into an active participant in the AI coding loop.

Matt Maisel’s Cedar-policy talk addresses a different layer: controlling what coding agents can do. The record states that coding agents have dangerous access to code and terminal, and that prompt injection makes soft guardrails unreliable [record_id:2345]. The proposed reference monitor uses Rust hooks and Cedar policies to deterministically intercept every shell command, file read, and other actions, with live-demo examples forbidding exfiltration and destructive behavior [record_id:2345]. This reflects a trend toward hard security boundaries around agent tools rather than relying solely on model behavior.

Piotr Ryciak’s “Vibe Check” talk provides the adversarial motivation for these controls. It describes vulnerability research across major AI-assisted IDEs and agents, including OpenAI Codex, Amazon Kiro, Google Antigravity, Cursor, and others [record_id:2335]. It organizes exploitation patterns by attacker effort and trigger model—zero-click, one-click, autorun behavior, and time-delayed execution—and promises a repeatable playbook and checklist for assessing and hardening AI-assisted IDE deployments [record_id:2335]. This suggests that the development environment itself has become a security boundary requiring systematic assessment.

### 4. Evaluation is becoming a core bottleneck

Another major trend is the shift from building agents to measuring whether they work. Joshua Saxe’s talk states this explicitly: as AI coding tools drive the cost of building security agents toward zero, the hard problem becomes knowing whether those agents work in the wild against real attacks and unseen vulnerabilities [record_id:2322]. The talk describes a progression from naive precision/recall on old data toward multi-dimensional evaluation that captures reasoning quality, evidence gathering, and tool-calling logic [record_id:2322]. It also connects measurement to improvement, saying proper measurement enables automated agent improvement using genetic algorithms and AI coding tools [record_id:2322].

The Netflix vulnerability discovery talk reinforces the evaluation concern from a practitioner angle. The abstract complains about LLMs “crying wolf about every string concatenation” and references closing 200 AI-generated findings [record_id:2374]. This is a strong signal that false positives and low-actionability outputs are major barriers to adoption. It also suggests that evaluation must consider developer workload and triage cost, not only whether a model can identify potential vulnerabilities.

The PyTorch optimization example provides a rare direct performance metric, but not an evaluation framework. It reports a dramatic speedup after AI-assisted profiling and implementation changes [record_id:2223]. The browser-extension demo reports partial live-demo failure before eventual success, which informally illustrates reliability challenges but does not provide systematic evaluation [record_id:2189].

### 5. AI-assisted development increases abstraction—and concern about skill erosion

The BSidesLV 2025 record is the strongest source on organizational and workforce risks. It warns that vibe coding can produce code that appears functional but fails security review, and it raises concerns about hiring pipelines being flooded with candidates lacking fundamental security understanding [record_id:2568]. It also warns that executives may develop unrealistic expectations about AI capabilities and that developers may become disconnected from “bare-metal computing principles,” weakening collective security posture [record_id:2568].

Other records are less explicit about skill erosion but imply continued need for human engineering judgment. Goldberg’s Godot talk emphasizes architecture, testing, and modular design to keep AI-generated code stable [record_id:2183]. Evron’s Chrome-extension talk required specialized developer and tester prompts to overcome implementation issues [record_id:2189]. Flake’s PyTorch optimization required identifying a GPU-to-CPU transfer bottleneck and applying a PyTorch-specific design pattern with forward hooks [record_id:2223]. These examples imply that successful AI-assisted development still depends on technical framing, debugging, review, and system design.

## Methods, Tools, And Approaches Discussed

The records mention a broad toolchain, including Claude Code, Cursor, Gemini, Gemini CLI, GPT-5, Godot, Godot MCP, DeepSeek, Pygame, Google Docs, Jira, Chrome extensions, PyTorch, MCP servers, Cedar policies, Rust hooks, GitHub Copilot CLI, and agentic vulnerability discovery pipelines.

One practical approach is interactive agentic implementation. In the Godot game record, Claude Code and Cursor are used to build a space shooter in the Godot engine, with attention to architecture, testing, and modular design for stability [record_id:2183]. In the Mail Goggles record, GPT-5 is used iteratively to build a Chrome extension, with separate developer and tester prompts used to address bugs, interface problems, and Manifest 3 issues [record_id:2189]. These examples show prompt-driven software creation, but also show that prompt design, testing, and iteration remain central.

Another approach is AI-assisted profiling and performance refactoring. The PyTorch record describes using Claude Code and Gemini CLI side by side to profile a slow visualization script, identify GPU-to-CPU transfer overhead, and implement PyTorch forward hooks so activation-pattern hashes are computed during the forward pass on the GPU [record_id:2223]. This is one of the most technically detailed records and illustrates AI tools supporting performance maintenance rather than greenfield code generation.

Requirements engineering is represented by a custom Gemini “gem.” The described prompt asks questions one at a time, tracks conflicts between requirements, reduces chattiness, and produces formal requirements documents with version control [record_id:2227]. The output can be exported to Google Docs and possibly used as input to vibe coding workflows [record_id:2227]. This suggests a method in which AI-generated code is grounded in a more disciplined requirements artifact.

Several approaches focus on security context and policy enforcement. Gupta’s MCP server injects security context into the AI coding process by retrieving threat models, security requirements, and OWASP guidance before generation, then verifying code after generation [record_id:2373]. Maisel’s reference-monitor approach uses Rust hooks and Cedar policies to intercept commands and file reads and enforce deterministic restrictions against exfiltration or destructive actions [record_id:2345]. Ryciak’s catalog of exploitation patterns across AI-assisted IDEs proposes a playbook and checklist for hardening deployments [record_id:2335].

For vulnerability discovery, Behrens and Cassel describe an agentic source-to-sink pipeline intended to improve LLM-driven first-party vulnerability discovery in real codebases [record_id:2374]. The emphasis is not merely on finding more issues but on improving accuracy and actionability, reducing noisy reports that waste engineering time [record_id:2374].

For security-agent evaluation, Saxe’s talk discusses multi-dimensional evaluation of reasoning quality, evidence gathering, and tool-calling logic, moving beyond simple precision and recall on stale datasets [record_id:2322]. It also mentions automated agent improvement through genetic algorithms and AI coding tools [record_id:2322]. This is a methodological contribution at the level of measuring and improving agents, rather than building a single development tool.

The Google and OpenAI records frame broader organizational methods. Google’s talk discusses evaluating emerging cyberattack capabilities and tools such as CodeMender for building safer software [record_id:2321]. OpenAI’s “Code Is Free” talk advocates engineering-first, zero-friction security additions using LLMs, including asking models for new security invariants [record_id:2325]. These records suggest that leading organizations are positioning AI coding not just as a developer productivity tool but as part of software security strategy.

For downstream researchers interested in AI-assisted performance engineering, this is a key record.

The Godot game record is representative of constructive vibe coding. It describes a real game-building effort using Claude Code and Cursor, with a goal of shipping on Steam, and emphasizes architecture, testing, and modular design [record_id:2183]. ## Gaps, Limits, And Open Questions

The corpus is rich in practitioner claims and conference-demo descriptions, but thin on reproducible detail. Most records are abstracts or short summaries. They rarely include code repositories, datasets, evaluation protocols, failure rates, user studies, or longitudinal measurements of productivity and quality. The PyTorch optimization record provides a concrete speedup claim [record_id:2223], but most other records do not quantify outcomes.

There is also limited comparative evidence across tools. Claude Code, Cursor, Gemini, Gemini CLI, GPT-5, GitHub Copilot CLI, OpenAI Codex, Amazon Kiro, Google Antigravity, and other tools are mentioned, but the records do not provide controlled comparisons of their effectiveness, reliability, safety, or developer experience [record_id:2183] [record_id:2189] [record_id:2223] [record_id:2335] [record_id:2345]. Downstream researchers should avoid inferring tool superiority from these summaries alone.

The records raise but do not fully answer the question of how AI-generated code quality changes over project lifetime. Goldberg’s Godot talk emphasizes architecture, testing, and modular design to keep AI-generated code stable [record_id:2183], while the BSidesLV talk warns that functional-looking generated code can fail basic security scrutiny [record_id:2568]. But there is no longitudinal evidence showing whether vibe-coded systems become harder to maintain, whether test suites offset this risk, or which architectural practices work best.

The security records propose several controls—MCP security context injection, Cedar policy reference monitoring, agentic vulnerability discovery, CodeMender-like automated defense, and model-assisted security invariants—but do not establish how these controls should be combined into a coherent secure development lifecycle [record_id:2321] [record_id:2325] [record_id:2345] [record_id:2373] [record_id:2374]. Open questions include whether policy enforcement should happen at the IDE, terminal, OS, repository, CI/CD, or model-tool interface layer; how to manage developer friction; and how to audit agent actions.

Agent evaluation remains a major open area. Saxe’s talk identifies the need for multi-dimensional evaluation of reasoning, evidence gathering, and tool use [record_id:2322], while the Netflix talk highlights noisy vulnerability findings [record_id:2374]. However, the corpus does not provide standard metrics, benchmark suites, or shared evaluation datasets for coding agents operating in realistic repositories. It also does not resolve how to measure agent safety against prompt injection, malicious repositories, or time-delayed triggers, issues raised in the AI IDE security-failures record [record_id:2335].

Organizational and educational impacts are underdeveloped. The BSidesLV record raises concerns about hiring pipelines, unrealistic executive expectations, and declining low-level understanding [record_id:2568], but the corpus contains no empirical studies of developer skill changes, hiring outcomes, or organizational adoption patterns. Similarly, the [un]prompted “Vibecoded” micro-talk record suggests community use of vibe coding to build conference tools but gives too little detail to analyze what was built, how reliable it was, or how participants evaluated success [record_id:2352].

Finally, there is little evidence about legal, licensing, compliance, accessibility, or supply-chain implications of AI-assisted software development. The records focus primarily on security, productivity, requirements, and tooling. Future research should examine how AI coding agents interact with open-source licenses, dependency provenance, regulated development environments, secure build pipelines, and audit requirements.
