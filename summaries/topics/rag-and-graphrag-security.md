# Topic: RAG and GraphRAG security

## Executive Summary

The seven records collectively portray RAG and GraphRAG security as a 2025 conference-driven research area focused on two parallel directions: using retrieval-augmented systems to improve security operations, and securing the retrieval, embedding, and data pipelines that make those systems useful. Several talks frame RAG as an enabling architecture for SOC automation, cloud security reasoning, compliance mapping, alert triage, and explainable cyber decision support [record_id:118] [record_id:2515]. At the same time, other records emphasize that RAG pipelines introduce new attack surfaces around vector databases, embeddings, context poisoning, indirect prompt injection, data exposure, and manipulation of trusted AI outputs [record_id:1930] [record_id:1943] [record_id:2110] [record_id:2286] [record_id:2313].

The strongest theme is that retrieval changes the security boundary of LLM applications. Sensitive content may reside not only in prompts or model weights, but also in embeddings, vector stores, indexed documents, knowledge graphs, policy repositories, logs, and other connected context sources [record_id:1943] [record_id:2515]. The records repeatedly suggest that attackers can exploit this context layer through manipulation, poisoning, or extraction, while defenders can use the same context layer to ground AI outputs in real-world infrastructure and security telemetry [record_id:118] [record_id:1930] [record_id:2110].

The evidence base is mostly talk abstracts rather than papers, empirical evaluations, or detailed implementation reports. It is strong in identifying relevant risks and use cases, but thin on concrete mitigations, measurements, and comparative security outcomes. The topic is therefore best understood as an emerging applied security area: rich in practitioner concern and proof-of-concept framing, but still lacking shared benchmarks, formal threat models, and mature operational guidance.

## Research Landscape

The included records come from 2025 security and AI/security conferences: CAMLIS, DEF CON 33, BSidesSF, and BSidesLV. They are primarily talk or paper descriptions, not full technical artifacts. Two records are primary RAG/GraphRAG security entries: a CAMLIS paper on a GraphRAG-inspired approach to cloud infrastructure reasoning [record_id:118], and a BSidesLV talk on using RAG plus the Model Context Protocol for cybersecurity defense workflows [record_id:2515]. The remaining records are secondary-topic matches where RAG appears as part of broader AI application security, privacy, incident response, prompt injection, or model security discussions [record_id:1930] [record_id:1943] [record_id:2110] [record_id:2286] [record_id:2313].

The landscape is dominated by practitioner-oriented security talks rather than academic work. The records discuss real-world AI vulnerabilities, bug bounty tactics, incident response, Microsoft Copilot manipulation, privacy leakage, vector database risks, and application security of GenAI components. This suggests that RAG security is not treated as a standalone discipline in all venues; rather, it appears as a cross-cutting concern in AI security, privacy, cloud security, SOC automation, and application security.

A recurring pattern is that the same architectural feature that makes RAG useful—the ability to connect LLMs to external, current, domain-specific data—is also the source of security risk. RAG systems ingest and retrieve from data sources such as cloud configuration data [record_id:118], policy and threat intelligence repositories [record_id:2515], enterprise content and limited data sources in Copilot-like systems [record_id:2110], and vector databases containing embedded representations of sensitive information [record_id:1943]. The records imply that securing RAG requires attention to the entire retrieval pipeline, not just the model interface.

GraphRAG appears less frequently than general RAG. The clearest GraphRAG-related record is the CAMLIS entry on “Relational Inference GraphRAG,” which describes transforming cloud configuration data into a “security-enriched knowledge graph” to support natural-language reasoning about deployed infrastructure [record_id:118]. Other records discuss vector databases, agents, LLMs, and retrieval workflows, but do not specifically describe graph-based retrieval architectures [record_id:2313] [record_id:2515]. This means GraphRAG security is represented, but the evidence base is narrow.

## Major Themes And Trends

### RAG as a security operations enabler

Two records present RAG or GraphRAG as a positive capability for defenders. The CAMLIS paper introduces “Relational Inference GraphRAG,” or RIG-RAG, as an “LLM-assisted pipeline” that turns cloud configuration data into a “security-enriched knowledge graph” for natural-language reasoning about deployed infrastructure [record_id:118]. This positions GraphRAG as a way to make cloud security operations more agentic and context-aware. The emphasis is not on attacking RAG, but on using graph-structured infrastructure context to improve reasoning and operational security workflows.

Similarly, the BSidesLV talk “RAG Against the Machine” presents CADDIE as a RAG engine driven by the Model Context Protocol. It is intended to “supercharge SOCs, auditors, and compliance teams” by injecting “real-time policy, threat intel, and log data” into LLMs for gap analysis, alert triage, and regulatory mapping [record_id:2515]. This record frames RAG as a route toward explainable AI and better-grounded security automation, explicitly contrasting a “precise, compliant tool” with a “hallucinating risk vector” [record_id:2515].

Together, these records show a constructive trend: security teams are interested in RAG because it can connect LLMs to operational truth sources. The envisioned uses include cloud infrastructure reasoning, SOC triage, compliance mapping, audit support, and policy-aware automation [record_id:118] [record_id:2515]. However, these records mostly describe intended architectures and use cases, not measured effectiveness or deployed security outcomes.

### Retrieval as a new attack surface

A second major theme is that RAG creates attack surfaces around retrieved context, indexed data, and data-source trust. The DEF CON bug bounty talk explicitly lists “context poisoning” and “RAG manipulation” alongside adversarial prompts and indirect prompt injection as real-world AI vulnerabilities [record_id:1930]. This places RAG security in the same practical exploit ecosystem as prompt injection and application security testing. The talk description says traditional defenses often fail and promises “attack ideas” and strategies for finding high-impact AI flaws [record_id:1930], suggesting that retrieval manipulation is already being treated as bug-bounty-relevant.

The Copilot-focused record adds a concrete manipulation pattern: “data voids” can be hijacked when an AI system relies on limited data sources [record_id:2110]. The abstract says attackers can inject persistent malicious content, associate it with legitimate Microsoft topics, and exploit the AI system’s failure to validate key terms [record_id:2110]. This is not labeled as RAG in the raw text, but it is closely related to retrieval trust: a system draws from limited or weakly validated source material, and attackers influence the content that gets surfaced as authoritative guidance. The described impact is severe: Copilot could deliver dangerous installation instructions for command-and-control beacons, potentially guiding enterprise users toward compromised actions [record_id:2110].

The BSidesSF application security record broadens this attack-surface view by naming “agents, vector databases, and LLMs” as interconnected GenAI components that introduce complex security risks [record_id:2313]. Its framing—“not just prompt injection”—is important because it moves beyond model-only or prompt-only security toward component and workflow security. In RAG systems, vector databases and data workflows are often core infrastructure, so this record supports the trend that RAG security needs architectural controls, data protection, and secure agent interaction design [record_id:2313].

### Data leakage from embeddings, vector databases, and RAG workflows

Privacy and data exposure are central in the DEF CON talk on “shadow data.” The record says modern AI apps, especially those using LLMs and RAG workflows, carry hidden risks because sensitive data such as PII and social security numbers can be extracted through real-world attacks [record_id:1943]. It specifically mentions model inversion attacks against fine-tuned models and embedding inversion attacks on vector databases [record_id:1943]. This is one of the clearest records linking RAG infrastructure to data leakage: even if raw documents are not directly exposed, embeddings and vector stores may encode enough information to support sensitive-data extraction.

This record also challenges conventional privacy tooling. It states that PII scanning tools fail to recognize “the rich data that lives in these systems” and characterizes AI ecosystems as a potential privacy disaster [record_id:1943]. The implication is that organizations cannot rely only on scanning source documents or prompts; they must consider derived representations, indexes, vector embeddings, and retrieval outputs as sensitive assets. This theme overlaps with the incident response record, which mentions risks of data exposure in AI systems and a case involving a RAG-enabled LLM [record_id:2286].

The evidence here is conceptually strong because multiple records mention data exposure, vector databases, RAG workflows, and privacy risks [record_id:1943] [record_id:2286] [record_id:2313]. It remains thin on exact attack procedures, because the records are abstracts rather than full papers. Still, the recurring emphasis indicates that leakage from RAG-adjacent storage and retrieval layers is a major concern for downstream research.

### Incident response and operational visibility gaps

The BSidesSF incident response talk identifies practical challenges in responding to AI incidents: “insufficient logging, visibility, and accountability,” along with data exposure and prompt injection risks [record_id:2286]. It specifically says the talk examines a case of a “RAG-enabled LLM” and proposes triaging strategies and improved incident response practices [record_id:2286]. This record is significant because it shifts the discussion from prevention and exploitation to detection, response, and post-incident handling.

In RAG systems, incident response is likely complicated by the number of moving parts: user prompts, retrieved documents, retrieval rankings, embeddings, vector database state, model outputs, agent actions, and downstream tool calls. The record does not enumerate all of these, but its emphasis on logging and visibility gaps is directly relevant [record_id:2286]. If responders cannot reconstruct what context was retrieved, why it was retrieved, whether it was poisoned, or how it shaped the final answer, then it becomes difficult to prove root cause or scope.

This operational theme also connects to the defensive RAG records. CADDIE aims to inject log data, threat intelligence, and policy into LLMs for SOC and compliance workflows [record_id:2515]. RIG-RAG aims to support natural-language reasoning about deployed cloud infrastructure [record_id:118]. Both would require careful auditability if used in production security operations. The incident response record suggests that without improved visibility and accountability, RAG-enabled security tools may themselves become difficult to investigate when they fail or are attacked [record_id:2286].

### Trust, authority, and grounding failures

Several records implicitly or explicitly focus on misplaced trust in retrieved or generated content. The Copilot talk is the clearest: attackers can manipulate AI-generated responses by exploiting limited data sources, associating malicious content with legitimate Microsoft topics, and taking advantage of failures to validate key terms [record_id:2110]. This is a trust-boundary problem: the AI system appears to provide authoritative guidance, but the underlying data association has been corrupted.

The RAG + MCP defensive talk also frames grounding as both an opportunity and a risk. It proposes injecting real-time policy, threat intelligence, and log data into LLMs to make GenAI “precise” and “compliant,” rather than hallucination-prone [record_id:2515]. This assumes that trusted context can improve output quality. But the broader set of records warns that if the context is poisoned, incomplete, sensitive, or poorly governed, grounding may amplify rather than reduce risk [record_id:1930] [record_id:1943] [record_id:2110].

The application security record reinforces that GenAI systems are composed of agents, vector databases, and LLMs, and that security strategies must cover interactions, models, and data workflows [record_id:2313]. In other words, trust cannot be assigned simply to the model or the final output; it must be managed across the data supply chain that feeds retrieval.

## Methods, Tools, And Approaches Discussed

The records describe several architectural and security approaches, though most are high-level.

One notable architecture is RIG-RAG, or Relational Inference GraphRAG. It is described as an LLM-assisted pipeline that transforms cloud configuration data into a “security-enriched knowledge graph” [record_id:118]. The intended method is to use graph representation of cloud infrastructure to support natural-language reasoning about deployed infrastructure. This suggests a workflow where cloud configuration is ingested, transformed into relational graph form, enriched with security semantics, and queried through an LLM-enabled interface [record_id:118]. The record does not provide implementation details, but the approach is representative of GraphRAG’s potential in cloud security: make relationships among resources, configurations, and risks explicit for reasoning.

Another defensive architecture is CADDIE, described as a RAG engine driven by the Model Context Protocol [record_id:2515]. It uses RAG plus MCP to inject “real-time policy, threat intel, and log data” into LLMs [record_id:2515]. The targeted workflows include gap analysis, alert triage, and regulatory mapping. This record highlights a broader trend toward connecting LLMs to live operational data and governance context. It also suggests that explainability and compliance are selling points for RAG in security operations [record_id:2515].
